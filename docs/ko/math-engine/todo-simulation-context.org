* 목적

BYUL에서 게임은 제한된 시간 예산 안에서 수행되는 simulation이다. 게임과 공학용
simulation을 별도 알고리즘과 API로 나누지 않고 같은 core에 두 가지 실행 preset을
적용한다.

#+begin_src text
Simulation Core
  numal + numeq + geometry + units
       ^
simulation context
  speed preset / precision preset
       ^
game runtime / engineering / robotics / aerospace / offline analysis
#+end_src

* 기본 원칙

- 게임용·공학용 함수 복제 금지
- mode는 알고리즘 이름이 아니라 context preset
- 호출자는 preset 이후 필요한 항목만 override 가능
- 지원하지 않는 조합은 silent fallback 없이 상태 반환
- 시간 예산과 오차 예산을 모두 명시
- 동일 입력과 context는 문서화된 결과 계약을 가짐
- mode 전환 시 state의 단위, 좌표계와 ownership 유지

* Speed mode

대상:

- game runtime과 30/60/120 Hz tick
- 대량 entity, projectile와 particle
- 실시간 controller와 hardware-in-the-loop
- broad phase와 후보 탐색

기본 정책 후보:

- =f32= 또는 명시적 실시간 backend
- fixed step, semi-implicit Euler와 Verlet
- 작은 iteration/subdivision/evaluation budget
- 근사 predicate와 fast geometry path
- 고정 buffer, pool과 arena
- SIMD/GPU와 parallel batch 허용
- deadline 안의 best available result와 품질 상태

* Precision mode

대상:

- 공학·과학 검증
- offline trajectory와 replay audit
- CAD intersection과 geometry refinement
- 기준 결과와 회귀 oracle

기본 정책 후보:

- =f64= 기본, =f128=/multiprecision 선택
- RK4와 adaptive integrator
- 엄격한 absolute/relative/angular tolerance
- 큰 iteration/subdivision/evaluation budget
- robust/adaptive predicate와 collision refinement
- 오차, 잔차, 수렴과 평가 통계
- 필요시 동적 또는 caller-provided 확장 workspace

* 게임 내 혼합 실행

게임 전체를 한 mode로 고정하지 않는다.

#+begin_src text
world tick                 speed
collision broad phase      speed
ambiguous narrow phase     precision refinement
projectile runtime         speed
offline impact prediction precision
NPC movement               speed
replay validation          precision
graph preview              speed
idle refinement            precision
#+end_src

speed 결과가 애매하거나 중요도가 높을 때 precision query로 승격할 수 있다. 승격 조건,
최대 비용과 실패 시 정책은 호출자가 정한다.

* Context 후보

#+begin_src c
typedef enum byul_simulation_mode_t {
    BYUL_SIMULATION_SPEED,
    BYUL_SIMULATION_PRECISION
} byul_simulation_mode_t;

typedef struct byul_simulation_context_t {
    byul_simulation_mode_t mode;
    byul_numeric_backend_t numeric_backend;
    double absolute_tolerance;
    double relative_tolerance;
    double angular_tolerance;
    uint32_t max_iterations;
    uint32_t max_subdivisions;
    uint64_t evaluation_budget;
    uint64_t memory_budget;
    bool robust_geometry;
    bool adaptive_step;
    bool allow_simd;
    bool allow_gpu;
    bool deterministic;
} byul_simulation_context_t;
#+end_src

public 구조체로 확정하기 전 ABI 확장 전략을 검토한다. size/version field가 있는 구조와
opaque context plus getter/setter 방식을 비교한다.

* 결정론

결정론은 speed/precision과 독립된 정책이다.

#+begin_src text
speed + deterministic       lockstep, replay
speed + parallel            particle와 visual simulation
precision + deterministic   regression과 audit
precision + parallel        offline batch
#+end_src

- 연산과 reduction 순서
- thread scheduling과 GPU 결과
- FMA, denormal과 compiler fast-math
- RNG seed와 stream
- serialization과 backend version
- platform별 허용 오차와 bitwise 계약 구분

* 모듈별 적용

** numal/mathfunc

- numeric backend와 rounding
- fast approximation 허용 여부
- non-finite와 overflow 정책

** numeq

- solver tolerance와 iteration
- integrator와 adaptive step
- evaluation budget과 cancellation

** geometry

- tolerance와 robust predicate
- broad/narrow phase와 refinement
- GJK/EPA iteration과 contact manifold 크기

** expression/plot

- instruction과 recursion budget
- graph initial sample과 refinement
- CAS rewrite와 expression growth

** batch/GPU

- offload threshold
- deterministic reduction 여부
- CPU fallback과 결과 tolerance

* 금융과 CAD의 추가 정책

다섯 대표 분야 중 금융과 CAD는 speed/precision만으로 충분하지 않다.

- 금융: decimal exactness, scale, quantize와 rounding policy
- CAD: geometric robustness, topology tolerance와 healing policy

내부 context는 다음 네 축을 조합할 수 있어야 한다.

#+begin_src text
execution speed
numeric precision
decimal exactness
geometric robustness
#+end_src

사용자-facing preset은 단순하게 유지하되 module-specific policy를 별도로 확장한다.

* 테스트 기준

- 같은 API의 speed/precision 결과 비교
- speed 결과 오차가 선언된 범위 안에 있음
- precision mode의 잔차·수렴·오차 추정
- speed에서 precision으로 승격한 collision/trajectory 일관성
- deadline, cancellation과 budget exhaustion
- deterministic preset의 replay
- CPU/SIMD/GPU 결과 계약
- f32/f64와 high-precision oracle
- Windows, Linux와 macOS

* 완료 기준

- 게임과 공학 simulation이 동일 API를 사용한다.
- speed/precision preset과 override 규칙이 문서화된다.
- mode별 숫자, 적분기, geometry와 budget 기본값이 정의된다.
- 계산별 mode 혼합과 precision 승격이 가능하다.
- 결정론이 mode와 독립적으로 설정된다.
- 금융 decimal과 CAD robustness 정책을 별도 축으로 결합할 수 있다.
- 지원하지 않는 조합이 명시적 상태로 보고된다.

* 진행 조건

첫 구현은 모든 모듈을 context 하나에 결합하는 작업이 아니다. 공통 mode enum,
tolerance, iteration/evaluation budget과 cancellation 계약을 먼저 정의하고 기존
solver·integrator·collision API에 adapter로 적용한다. public header를 수정할 때는
BYUL header 선언·comment·license 규칙을 먼저 읽고 따른다.
