#+TITLE: BYUL Public C Header 선언 규칙
#+LANGUAGE: ko
#+OPTIONS: toc:3 num:t

* 문서 목적

이 문서는 BYUL public C header의 생성과 유지에 적용하는 선언 규칙을 정의한다.
Header 선언만으로 C ABI 구조와 기본 메모리 동작을 판별할 수 있어야 한다.

추가 의미 정보는 [[file:header-comment-rules.org][BYUL Public C Header 코멘트 규칙]]을 따른다.
Wrapper 변환 규칙은 [[file:wrapper-rules.org][BYUL Python Wrapper 자동 생성 규칙]]을 따른다.

* 적용 범위

이 규칙은 새 header와 기존 header 모두에 적용한다. Public header는 다음 요소를
갖춰야 한다.

- 표준 라이선스 블록과 파일 수준 Doxygen 문서
- 일관된 파일명, include guard, include 순서와 C linkage
- C ABI 호환 type과 함수 선언
- 이름으로 판별 가능한 allocation, ownership, lifetime 관계
- public type의 ABI 크기와 layout 정보
- 모든 public 선언의 Doxygen 문서와 필요한 =@byul.*= metadata

Doxygen 형식, NULL, 오류, thread, callback 등 선언만으로 알 수 없는 계약은
[[file:header-comment-rules.org][Header 코멘트 규칙]]에서 정의한다.

* 표준 라이선스 블록

모든 BYUL 작성 header는 첫 번째 줄부터 다음 블록을 MUST 포함한다. 복사의 유일한
기준 원본은 [[file:header-license-block.txt][header-license-block.txt]]다. 문서의 예제를 다시 입력하지 말고
기준 파일의 내용을 그대로 복사한다.

#+begin_src c
/*
 * Copyright (c) 2025-2026 ByulPapa (byuldev@outlook.kr)
 * This file is part of the Byul World project.
 * Licensed under the Byul World Source-Available Non-Commercial License v1.0 (2025).
 * See the LICENSE file in the project root for full license terms.
 */
#+end_src

적용 규칙은 다음과 같다.

- 라이선스 블록 앞에는 빈 줄, 다른 comment, include guard 또는 선언을 두지 않는다.
- UTF-8 BOM을 새로 추가하지 않는다. 파일은 첫 번째 바이트의 =/*= 로 시작한다.
- 블록 뒤에는 빈 줄 하나를 두고 파일 수준 Doxygen 문서를 작성한다.
- 파일별로 저작권자, 연도, 프로젝트명, 라이선스명을 임의 변경하지 않는다.
- 축약명 =Byul World Public License= 를 사용하지 않는다. 저장소 =LICENSE= 의 정식
  명칭을 사용한다.
- 사용자 정의 라이선스이므로 공식 SPDX identifier가 정해지기 전에는 임의의
  =SPDX-License-Identifier= 를 추가하지 않는다.
- 전체 법적 조건의 기준은 저장소 루트의 =LICENSE= 다. 이 짧은 블록이 =LICENSE= 를
  대체하지 않는다.
- 외부에서 가져온 third-party 또는 vendor header의 기존 라이선스 notice를 삭제하거나
  이 블록으로 덮어쓰지 않는다. 해당 파일은 원 라이선스와 =THIRD_PARTY_NOTICES.md= 를
  따른다.
- BYUL generator가 생성하는 header도 BYUL 작성 산출물이면 generator template에 이
  블록을 넣어 매번 동일하게 출력한다.

Header lint는 BYUL 작성 header의 첫 줄부터 기준 파일과 정확히 같은 블록인지 검사한다.
줄바꿈 형식은 플랫폼에 맞출 수 있지만 문자, 공백, 줄 순서가 달라지면 오류다.

* 파일 수준 Doxygen 문서

모든 BYUL 작성 public header는 라이선스 블록 바로 아래에 파일의 역할을 소개하는
Doxygen block을 MUST 둔다. 이 블록의 정식 명칭은 "파일 수준 Doxygen 문서"다.

#+begin_src c
/**
 * @file coord.h
 * @brief Declares the public C ABI for two-dimensional coordinates.
 *
 * Provides the coordinate value type and operations used to create, inspect,
 * compare, and measure grid coordinates across the BYUL public ABI.
 */
#+end_src

필수 작성 규칙은 다음과 같다.

- =@file= 값은 확장자를 포함한 실제 header 파일명과 정확히 같아야 한다.
- =@brief= 는 header가 외부에 제공하는 핵심 책임을 영어 한 문장으로 설명한다.
- 빈 줄 뒤의 상세 설명은 대표 type, 제공 기능, 모듈 경계를 1개 이상의 문장으로
  설명한다.
- 구현 알고리즘이나 변경 이력보다 caller가 이 header를 include하는 이유를 쓴다.
- 저작권자, 라이선스, 작성일, 수정일은 반복하지 않는다. 이 정보는 표준 라이선스
  블록과 버전 관리 기록이 담당한다.
- 함수별 allocation, NULL, lifetime 계약은 파일 소개에 몰아넣지 않고 해당 선언의
  Doxygen와 =@byul.*= tag에 작성한다.
- =@file= 블록은 문서 생성용이며 C ABI나 wrapper method를 생성하지 않는다.

Header의 시작 순서는 반드시 다음과 같다.

#+begin_example
표준 라이선스 블록
빈 줄 하나
파일 수준 Doxygen 문서
빈 줄 하나
include guard
#+end_example

Header lint는 =@file= 존재 여부, 실제 basename과의 일치, =@brief= 및 상세 설명의
존재 여부를 검사한다.

* 규칙 우선순위

1. 실제 C ABI 선언
2. 이 문서의 전역 명명 및 메모리 규칙
3. 구조화된 header comment metadata
4. 일반 설명문

Comment가 선언과 충돌하면 선언 오류로 처리한다. Generator가 comment를 믿고 잘못된
C signature를 보정해서는 안 된다.

* Public header 기본 형식

** 파일명

파일명은 소문자 snake_case를 사용한다.

#+begin_example
coord.h
coord_list.h
route_finder.h
dstar_lite.h
#+end_example

Header file stem은 대표 C type과 일치해야 한다.

#+begin_example
coord.h       -> coord_t
coord_list.h  -> coord_list_t
navgrid.h     -> navgrid_t
#+end_example

** Include guard

Include guard는 file stem의 대문자 형식을 사용한다.

#+begin_src c
#ifndef BYUL_COORD_LIST_H
#define BYUL_COORD_LIST_H

/* declarations */

#endif /* BYUL_COORD_LIST_H */
#+end_src

기존 guard와 충돌하지 않도록 모든 public header에 =BYUL_= prefix를 사용한다.

** Include 순서

순서는 다음과 같다.

1. 표준 C header
2. =byul_config.h=
3. BYUL dependency header

#+begin_src c
#include <stdbool.h>
#include <stddef.h>
#include <stdint.h>

#include "byul_config.h"
#include "coord.h"
#+end_src

사용하지 않는 header를 include하지 않는다. Public declaration에 필요한 type은
반드시 직접 include하거나 명시적으로 forward declaration한다.

** C linkage

모든 public 함수는 C++에서도 C linkage를 사용한다.

#+begin_src c
#ifdef __cplusplus
extern "C" {
#endif

/* public C ABI */

#ifdef __cplusplus
}
#endif
#+end_src

** Export macro

모든 shared library public function에는 =BYUL_API=를 적용한다.

#+begin_src c
BYUL_API coord_t* coord_create(void);
#+end_src

Public type 선언에는 platform export attribute를 붙이지 않는다.

* C ABI 허용 타입

** 허용

- =void=
- =bool=
- 고정폭 정수: =int8_t=, =uint32_t= 등
- ABI에서 크기를 고정한 =int=, =unsigned int=
- =float=, =double=
- =size_t=, =uintptr_t=
- C enum
- C struct
- opaque struct pointer
- function pointer typedef
- =char*=와 =const char*=

** 금지

- C++ class
- =std::string=
- =std::vector=
- template type
- reference type
- overloaded function
- default argument
- exception이 ABI 밖으로 나가는 함수
- compiler별 layout이 다른 비표준 type

** 정수 크기

파일 형식, network, hash, serialization에 노출되는 정수는 fixed-width type을
사용한다. 단순 count/index가 =int=를 사용한다면 전체 API에서 같은 범위 규칙을
적용한다.

* Type 선언 규칙

** Type 저장 방식 분류

모든 public type은 선언 구조에 따라 다음 중 하나로 분류 가능해야 한다.

| 분류 | 선언 형태 | storage 생성 | 해제 |
|------+-----------+--------------+------|
| basic-value | 기본 자료형만 가진 public struct | caller storage | 불필요 |
| resource-value | public struct + 내부 resource | caller storage + init | free |
| opaque-object | opaque struct | create/copy | destroy |
| view | =*_view_t= public struct | 값 복사 | 불필요 |

=basic-value= 판정은 재귀적으로 수행한다. 다음 field만 허용한다.

- 정수와 고정폭 정수
- =bool=
- =float=, =double=
- enum
- 다른 basic-value struct
- basic-value element의 고정 길이 배열

다음 중 하나라도 있으면 basic-value가 아니다.

- object pointer
- owning pointer
- function pointer
- flexible array member
- platform resource handle
- hidden allocator state

Owning pointer를 가진 public struct는 SHOULD NOT 사용한다. 내부 memory를 소유하는
object는 SHOULD opaque type으로 만든다. 외부 memory를 참조하는 public struct는
MUST =*_view_t= 이름을 사용한다.

** Opaque object

내부 layout이 public 계약일 필요가 없는 object는 opaque type으로 선언한다.

#+begin_src c
typedef struct s_coord_list coord_list_t;
#+end_src

Opaque type은 ABI 안정성 측면에서 기본 선택이다.

** Public value struct

호출자가 stack에 생성하거나 값으로 반환해야 하는 작은 data type만 public
layout을 허용한다.

#+begin_src c
typedef struct s_coord {
    int x;
    int y;
} coord_t;
#+end_src

Public struct 규칙:

- field 순서를 변경하면 ABI 변경이다.
- field type을 변경하면 ABI 변경이다.
- 중간에 field를 삽입하지 않는다.
- bit-field를 사용하지 않는다.
- pointer field의 소유권을 struct field만으로 노출하지 않는다.
- compiler packing pragma에 의존하지 않는다.

** View struct

View는 외부 memory를 참조하지만 소유하지 않는 public value struct다.

#+begin_src c
typedef struct s_coord_view {
    const coord_t* data;
    int length;
} coord_view_t;
#+end_src

규칙:

- type 이름은 MUST =*_view_t=로 끝난다.
- pointer가 가리키는 memory를 view가 해제하지 않는다.
- view 자체를 위한 create/destroy를 제공하지 않는다.
- pointer 유효 기간과 mutation invalidation은 comment 규칙에 명시한다.

* ABI 크기와 layout 정보

** sizeof 함수

Native source에서 object storage를 할당하거나 public struct storage를 사용하는 모든
type은 MUST =<type>_sizeof()=를 제공한다.

#+begin_src c
BYUL_API size_t coord_sizeof(void);
BYUL_API size_t coord_list_sizeof(void);
#+end_src

구현은 실제 native type을 기준으로 한다.

#+begin_src cpp
size_t coord_sizeof(void) {
    return sizeof(coord_t);
}

size_t coord_list_sizeof(void) {
    return sizeof(coord_list_t);
}
#+end_src

Opaque type의 =sizeof= 공개는 caller가 storage를 직접 할당해도 된다는 의미가
아니다. Opaque type은 계속 create/destroy로만 생성하고 해제한다. Opaque sizeof는
SDK ABI 검증, memory diagnostic, wrapper-DLL 일치 검사에만 사용한다.

** alignof 함수

=sizeof=를 제공하는 모든 type은 MUST =<type>_alignof()=도 제공한다.

#+begin_src c
BYUL_API size_t coord_alignof(void);
#+end_src

#+begin_src cpp
size_t coord_alignof(void) {
    return alignof(coord_t);
}
#+end_src

Python wrapper는 다음을 검증한다.

#+begin_src python
assert ffi.sizeof("coord_t") == C.coord_sizeof()
assert ffi.alignof("coord_t") == C.coord_alignof()
#+end_src

Opaque type은 CFFI에서 layout을 알 수 없으므로 =ffi.sizeof()=와 직접 비교하지 않고
SDK ABI manifest의 native size와 DLL 반환값을 비교한다.

** offsetof 함수

Public layout struct의 모든 field는 SDK ABI 정보에 offset을 기록해야 한다. Runtime
검증이 필요한 type은 =<type>_offsetof_<field>()=를 제공한다.

#+begin_src c
BYUL_API size_t coord_offsetof_x(void);
BYUL_API size_t coord_offsetof_y(void);
#+end_src

#+begin_src cpp
size_t coord_offsetof_x(void) {
    return offsetof(coord_t, x);
}
#+end_src

Opaque type에는 offsetof API를 제공하지 않는다.

** sizeof와 동적 memory 크기 구분

=<type>_sizeof()=는 struct/object 본체 크기만 반환한다. 내부에서 별도로 할당한
heap buffer 크기는 포함하지 않는다.

내부 heap을 소유하는 type은 SHOULD 다음 query를 제공한다.

#+begin_src c
BYUL_API size_t coord_list_get_memory_size(
    const coord_list_t* list);
#+end_src

의미:

#+begin_example
<type>_sizeof()          고정 object storage 크기
<type>_get_memory_size() 현재 object가 직접 소유한 전체 native memory 크기
#+end_example

=get_memory_size=가 allocator overhead, shared allocation, child object memory를
포함하는지는 comment 규칙에 정확히 명시한다. 같은 module의 함수끼리 계산 범위를
다르게 정의하면 안 된다.

** Enum 크기

Public enum은 SDK ABI manifest에 =sizeof(enum-type)=와 모든 상수 값을 기록한다.
Runtime 검증이 필요하면 =<enum-name>_sizeof()=를 제공한다.

Enum 값은 기존 ABI에서 변경하지 않는다.

** 기본 ABI 환경 정보

SDK는 최소한 다음 정보를 manifest에 기록해야 한다.

- =sizeof(bool)=
- =sizeof(int)=
- =sizeof(long)=
- =sizeof(size_t)=
- =sizeof(void*)=
- byte order
- platform과 architecture
- toolchain과 configuration
- runtime library 종류

Public ABI에서는 platform별 크기가 달라지는 =long=보다 fixed-width integer를
우선한다.

** ABI version과 fingerprint

Library semantic version과 ABI version을 분리한다.

#+begin_src c
BYUL_API uint32_t byul_abi_version(void);
BYUL_API uint64_t byul_abi_fingerprint(void);
#+end_src

ABI fingerprint에는 다음이 반영되어야 한다.

- public type 이름
- sizeof와 alignof
- public field type과 offsetof
- enum 이름과 값
- public 함수 signature
- callback signature
- public 상수

Wrapper는 생성에 사용한 SDK fingerprint와 runtime DLL fingerprint가 다르면 import를
실패시켜야 한다.

* Allocation과 deallocation 연결

** 할당 함수의 해제 쌍

새 memory를 반환하는 모든 public 함수는 명명 규칙으로 대응 해제 함수를 결정할 수
있어야 한다.

#+begin_example
coord_create       -> coord_destroy
coord_copy         -> coord_destroy
route_create       -> route_destroy
route_copy         -> route_destroy
coord_copy_string  -> byul_string_destroy
#+end_example

BYUL에서 할당한 memory를 caller가 C runtime =free()=로 해제하게 하면 안 된다.
특히 Windows CRT 경계를 넘는 memory는 반드시 BYUL export 함수로 해제한다.

** Array allocation

Array를 할당해 반환하는 함수는 다음 정보를 선언과 comment로 제공한다.

- element type
- element count 반환 위치
- array 해제 함수
- element별 destroy 필요 여부

#+begin_src c
BYUL_API coord_t* route_copy_coords(
    const route_t* route,
    int* out_count);

BYUL_API void coord_array_destroy(
    coord_t* coords,
    int count);
#+end_src

가능하면 array allocation 반환보다 caller buffer =export/fetch= API를 우선한다.

** Reallocation과 pointer invalidation

Container mutation으로 내부 buffer가 재할당될 수 있다. 다음 동작은 기존
=get/peek/front/back= pointer를 무효화할 수 있으므로 comment 규칙에서 MUST
명시한다.

#+begin_example
push
insert
remove
clear
reserve
resize
sort
#+end_example

Generator는 invalidation 정보를 바탕으로 반환 child wrapper의 안전 정책을
결정한다. 정보가 없으면 내부 pointer 반환 API 자동 생성을 거부한다.

* 값 초기화와 복사 특성

** Zero-valid

Public struct는 모든 bit가 0인 상태가 유효한지 comment metadata로 명시한다.

#+begin_example
zero-valid
init-required
#+end_example

=ffi.new()=가 memory를 0으로 초기화하더라도 =init-required= type은 반드시 native
=init= 함수를 호출한다.

** Copy semantics

Type은 다음 중 하나로 분류한다.

#+begin_example
trivial-copy
deep-copy
non-copyable
#+end_example

- basic-value는 기본적으로 =trivial-copy= 후보지만 comment와 lint로 확인한다.
- resource type은 =assign/copy= 함수가 있을 때만 =deep-copy= 가능하다.
- mutex, thread, callback execution state를 가진 object는 기본적으로
  =non-copyable=이다.
- copy 함수가 없는 type에 Python =copy()=를 자동 생성하지 않는다.

** Type 이름

#+begin_example
struct tag: s_<name>
typedef:    <name>_t
enum tag:   e_<name>
callback:   <purpose>_func
#+end_example

예:

#+begin_src c
typedef struct s_route route_t;
typedef enum e_route_dir route_dir_t;
typedef void (*move_func)(const coord_t* coord, void* userdata);
#+end_src

* Enum 규칙

Public enum은 tag와 typedef 이름을 모두 명시한다.

#+begin_src c
typedef enum e_route_dir {
    ROUTE_DIR_UNKNOWN = 0,
    ROUTE_DIR_RIGHT = 1,
    ROUTE_DIR_UP_RIGHT = 2
} route_dir_t;
#+end_src

규칙:

- 첫 값은 가능한 경우 =*_UNKNOWN = 0= 이다.
- ABI에 노출된 값은 명시적인 정수를 사용한다.
- 기존 값의 숫자를 변경하지 않는다.
- 새 값은 뒤에 추가한다.
- sentinel count가 필요하면 =*_COUNT=를 마지막에 둔다.
- flag enum은 일반 enum과 구분되는 이름 및 bit value를 사용한다.

* 함수명 기본 규칙

함수명은 다음 형식이다.

#+begin_example
<대표-type-name>_<operation>
#+end_example

#+begin_example
coord_create
coord_get_x
coord_set_x
coord_distance
coord_list_get
route_finder_run
#+end_example

대표 type과 관련 없는 module function은 module prefix를 사용한다.

함수명은 Python wrapper의 메모리 판단과 이름 변환 입력이므로 동의어를 임의로
섞지 않는다.

* Object 생명주기 함수

** create

#+begin_src c
BYUL_API coord_t* coord_create(void);
BYUL_API coord_t* coord_create_full(int x, int y);
#+end_src

의미:

- heap에 새 object를 할당한다.
- caller가 소유한다.
- 대응 =destroy=가 있다.
- 실패 시 NULL을 반환한다.

Variant 이름은 구체적인 생성 의미를 나타낸다. 단순히 parameter가 많다는 이유만으로
=full=을 남용하지 않고 프로젝트 전체에서 일관되게 사용한다.

** init

#+begin_src c
BYUL_API int coord_init(coord_t* out);
BYUL_API int coord_init_full(coord_t* out, int x, int y);
#+end_src

의미:

- caller가 준비한 storage를 초기화한다.
- object memory를 할당하지 않는다.
- 내부 resource를 할당할 수 있다.
- 성공/실패 status 규칙을 따른다.

** free

#+begin_src c
BYUL_API void type_free(type_t* object);
#+end_src

의미:

- =init=된 object의 내부 resource를 정리한다.
- =object= storage 자체는 해제하지 않는다.

** destroy

#+begin_src c
BYUL_API void coord_destroy(coord_t* coord);
#+end_src

의미:

- create/copy/clone이 만든 object를 해제한다.
- 내부 resource와 object storage를 모두 정리한다.
- NULL 입력 허용 여부는 전역 정책 또는 comment 규칙으로 명시한다.

** assign

#+begin_src c
BYUL_API int coord_assign(coord_t* dst, const coord_t* src);
#+end_src

의미:

- 이미 유효한 =dst=에 =src= 값을 깊은 복사한다.
- =dst= pointer 자체를 변경하거나 해제하지 않는다.
- self assignment를 안전하게 처리한다.

** copy와 clone

#+begin_src c
BYUL_API coord_t* coord_copy(const coord_t* src);
#+end_src

의미:

- 독립적인 새 object를 할당한다.
- 반환값은 caller 소유다.
- 대응 destroy가 필요하다.
- shallow copy가 필요하면 이름에 명시하며 기본 copy는 deep copy다.

* Property 함수

** getter

#+begin_src c
BYUL_API int coord_get_x(const coord_t* coord);
#+end_src

형식:

#+begin_example
<type>_get_<property>(const <type>_t* object)
#+end_example

Getter는 object를 변경하지 않으므로 첫 parameter에 =const=를 적용한다.

** setter

#+begin_src c
BYUL_API void coord_set_x(coord_t* coord, int value);
#+end_src

형식:

#+begin_example
<type>_set_<property>(<type>_t* object, <value-type> value)
#+end_example

Setter는 값을 복사한다. Pointer를 저장하거나 소유권을 가져가는 동작에 =set=을
사용하지 않는다.

** fetch property

복합 값을 out parameter로 복사할 때 =fetch=를 사용한다.

#+begin_src c
BYUL_API bool route_finder_fetch_start(
    const route_finder_t* finder,
    coord_t* out_start);
#+end_src

* 메모리 의미를 나타내는 함수명

** allocation 발생

다음 이름은 새 allocation을 반환한다.

#+begin_example
create
copy
clone
release
#+end_example

** allocation 없음

다음 이름은 새 allocation을 만들지 않는다.

#+begin_example
get
peek
find
contains
is
has
set
bind
#+end_example

Pointer 반환 =get/peek=는 object 내부 주소다.

** caller buffer 복사

#+begin_example
fetch
export
#+end_example

=fetch=는 정해진 단일 값 또는 caller buffer로 복사한다. =export=는 여러 값 또는
직렬화된 출력을 caller buffer로 복사한다.

** 소유권 이동

#+begin_example
take     caller -> callee
release  callee -> caller
#+end_example

소유권을 이동하는 함수에 =set=, =add=, =push=를 사용하면 안 된다.

** non-owning 보관

#+begin_example
bind
unbind
#+end_example

=bind=는 object 또는 callback pointer를 보관하지만 소유하지 않는다.

* Collection 함수 규칙

Collection은 다음 vocabulary를 사용한다.

#+begin_example
length
empty
get
front
back
push_back
pop_back
push_front
pop_front
insert
remove_at
remove_value
clear
reverse
contains
find
#+end_example

규칙:

- =get/front/back= pointer 반환은 내부 주소다.
- 값을 복사해 반환하려면 함수명에 =copy= 또는 =fetch=를 넣는다.
- =push/insert=는 기본적으로 값을 복사한다.
- pointer 소유권을 가져가면 =take=를 이름에 포함한다.
- =pop=의 반환이 value인지 pointer인지 API family 전체에서 통일한다.
- empty collection 결과는 반환형으로 명확히 표현한다.

* Query와 비교 함수

#+begin_example
<type>_equal
<type>_compare
<type>_hash
<type>_contains
<type>_is_<state>
<type>_has_<item>
#+end_example

규칙:

- =equal=은 bool을 반환한다.
- =compare=은 음수/0/양수 규칙을 사용한다.
- =hash=는 고정된 unsigned type을 사용한다.
- query 함수는 object를 변경하지 않으므로 가능한 모든 pointer에 =const=를 쓴다.

* 연산 함수 규칙

** out-parameter 연산

#+begin_src c
BYUL_API void coord_add(
    coord_t* out,
    const coord_t* a,
    const coord_t* b);
#+end_src

출력 parameter는 첫 번째이며 =out= 또는 =out_<name>=으로 명명한다.

** in-place 연산

#+begin_src c
BYUL_API void coord_iadd(coord_t* io, const coord_t* other);
#+end_src

In-place operation은 =i<operation>= naming을 사용하고 첫 parameter를 =io= 또는
=inout_<name>=으로 명명한다.

* Pointer parameter 규칙

Parameter 이름과 =const=가 의미를 가져야 한다.

- =const T*= 입력 전용
- =T*= + =out_*= 출력 전용
- =T*= + =io_/inout_*= 입력/출력
- non-const object first parameter는 method receiver

NULL 허용 여부는 선언만으로 완전히 표현할 수 없으므로 comment 규칙에서
구조화하여 명시한다.

* 문자열 함수 규칙

#+begin_example
get_<name>_string   내부/static 문자열
copy_<name>_string  새 allocation 문자열
fetch_<name>_string caller buffer 복사
#+end_example

Owned string은 통일된 BYUL 문자열 해제 함수를 사용한다. 반환된 문자열을 =free()=
또는 module별 임의 함수로 섞어 해제하면 안 된다.

모든 public text encoding은 UTF-8을 기본으로 한다.

* Buffer와 array 규칙

Caller 제공 buffer 함수의 기본 형식:

#+begin_src c
BYUL_API int route_export_coords(
    const route_t* route,
    coord_t* out_coords,
    int capacity);
#+end_src

규칙:

- object
- out buffer
- capacity
- 실제 작성 개수 반환

순서로 통일한다. 필요한 길이 조회 함수는 =get_<item>_count= 또는 =length= 규칙을
사용한다.

* Callback 선언 규칙

** Typedef

#+begin_src c
typedef void (*move_func)(
    const coord_t* coord,
    void* userdata);
#+end_src

Callback parameter 순서는 domain parameter 뒤에 =userdata=를 마지막으로 둔다.

** 반환 pointer

Callback이 pointer를 반환하면 이름에 allocation 의미를 포함한다.

#+begin_example
changed_coords_copy_func
changed_coords_get_func
#+end_example

=copy= callback 결과는 수신자가 destroy한다. =get= callback 결과는 callback
소유자에게 남아 있으며 수신자가 destroy하지 않는다.

** 등록 함수

#+begin_example
<type>_bind_<purpose>_callback
<type>_unbind_<purpose>_callback
#+end_example

함수 pointer와 userdata는 한 등록 함수에서 함께 전달한다. 별도의 setter로 나누지
않는다.

* 오류 반환 규칙

Status 반환은 project 전역 type 또는 다음 규칙으로 통일한다.

#+begin_example
0      성공
< 0    오류
> 0    status가 아닌 count/result일 때만 사용
#+end_example

Bool API는 성공/실패 또는 predicate 중 한 의미만 사용한다. 함수 family 안에서
의미를 섞지 않는다.

Pointer API는 NULL의 의미를 comment 규칙으로 명시한다.

* Parameter 이름 규칙

다음 이름을 통일해서 사용한다.

| 의미 | 이름 |
|------+------|
| method receiver | object 또는 구체 type 이름 |
| 입력 source | src |
| 출력 destination | dst 또는 out |
| 출력 값 | out_<name> |
| 입력/출력 값 | io 또는 inout_<name> |
| 배열 용량 | capacity |
| 실제 개수 | out_count |
| callback context | userdata |
| index | index |
| 길이 | length |

같은 의미에 =ctx=, =user=, =data=, =udata=를 섞지 않는다.

* Declaration 배치 순서

각 header는 다음 순서로 선언한다.

1. constant와 enum
2. forward/opaque type
3. public value struct
4. callback typedef
5. create/init
6. free/destroy
7. assign/copy
8. property get/set/fetch
9. collection/query
10. operation
11. callback bind/unbind
12. debug/print/string

* Deprecated API

Deprecated symbol을 즉시 제거하지 않는다. 선언은 유지하되 comment 규칙의
=@deprecated=를 사용하고 replacement symbol을 명시한다.

새 wrapper generator는 deprecated API를 별도 목록에 기록할 수 있지만 기존
호환성이 필요한 동안 CFFI 선언에서는 유지한다.

* 선언 lint 실패 조건

다음은 public header lint 오류다.

- 표준 라이선스 블록이 없거나 기준 원문과 다름
- 파일 수준 Doxygen 문서가 없거나 =@file= 값이 실제 파일명과 다름
- 필수 public 선언에 Doxygen 또는 필요한 =@byul.*= metadata가 없음
- file stem과 대표 type 불일치
- create가 있지만 destroy가 없음
- copy/clone pointer 반환인데 destroy가 없음
- pointer 반환 함수 이름으로 allocation 의미를 판별할 수 없음
- get/set property type 불일치
- non-const query receiver
- set 함수가 pointer를 저장하거나 소유권을 가져감
- callback pointer 반환 의미가 이름에 없음
- owned string에 해제 함수가 없음
- buffer 출력인데 capacity가 없음
- public C ABI에 C++ type이 노출됨
- header 선언과 DLL 구현 signature 불일치
