* 목적

=byul/balix= 안의 수학, 물리 법칙, 기하, 미분·적분, 보간, 제어와 추정 계산을
독립 Math Engine 소유로 옮긴다. Balix는 world와 객체 상태를 보유하고 tick마다
필요한 계산을 요청하며 결과를 event, response와 history에 적용하는 소비자다.

이 문서는 구현 계획이다. 현재 source를 이동하거나 public 함수를 변경하지 않았으며,
각 단계의 회귀 corpus와 ABI gate가 닫힌 뒤에만 다음 단계로 진행한다.

* 전체 Header 리팩터링과 공동 소유권

이 문서는 새 canonical Math Engine 공식·kernel·neutral value/result, public header/API와
그 source의 유일한 design owner다. 전체 header 리팩터링과 Simulation consumer TODO는
기존 Balix header의 ABI/layout/install, DTO/lifecycle, adapter와 legacy forwarding을
소유하며 같은 canonical Math API를 다시 설계하지 않는다.

기존 Balix header와 새 Math Engine header는 일대일 rename 관계가 아니다. 하나의 Balix
header가 기존 compatibility/adapter header로 남으면서 0개 이상의 Math Engine successor
header로 분리될 수 있다.
[[file:../header-refactor-current/header-role-manifest.json][header-role-manifest.json]]을
Runtime, Simulation과 Math Engine이 공유하는 유일한 ownership manifest로 사용하며
다음 관계를 그 record에 유지한다.

#+begin_src text
initial_asset:
  asset_id
  current_path
  current_owner
  disposition
  compatibility_path
  successor_headers[]

created_successor:
  canonical_path
  canonical_owner
  design_todo
  install_target
  export_target
#+end_src

각 symbol, source와 새 header의 canonical owner는 정확히 하나다. 이 문서는
=successor_headers[]=의 이름과 내용을 승인하고, Simulation TODO는 해당 경로를 참조하여
Balix compatibility mapping만 승인한다. 공동 inventory와 owner 승인이 끝나기 전에 두
문서가 독립적으로 successor 설계나 source 이동을 시작하지 않는다.

이 master와 연결된 세부 추출 TODO가 canonical 계산 구현으로 =DOING=인 동안
Simulation의 Balix numeric/function 단계는 =TODO= 또는 =BLOCKED=를 유지한다. 아래
canonical handoff gate를 통과한 asset만 Simulation이 기존
ABI/layout/lifecycle/legacy forwarding 합류 gate로 전환한다. Adapter 합류 결과까지
검증한 뒤 해당 세부 추출 TODO 전체를 =DONE=으로 닫는다. Simulation의 P0 lifecycle
수정처럼 canonical 수학 결과를 필요로 하지 않는 단계는 이 순서와 독립적으로 수행할
수 있다.

** Canonical handoff Gate

각 Balix leaf TODO는 다음 증거를 모두 제공해야 해당 asset의 handoff가 가능하다.

- 공식·상수·private helper와 public symbol이 단일 ownership manifest의 owner와
  =design_todo= 하나에 연결된다.
- Balix type/include가 없는 canonical header, neutral value/result/status와 계산
  source가 Math Engine component에서 build·test된다.
- Formula corpus, non-finite/degenerate/failure와 old semantic mapping이 검증된다.
- Legacy Balix forwarding/DTO adapter, aggregate/install/link 전환은 아직 consumer와
  Simulation 합류 gate의 책임으로 명시돼 있다.

* 분류 기준

시간, 속도, 가속도를 인자로 받는다는 이유만으로 Balix 계산이 되지 않는다. 동일
입력에서 외부 상태를 읽거나 바꾸지 않고 값 또는 명시적 result를 반환하는 단발 계산은
Math Engine이다.

| 책임 | Math Engine | Balix |
|------+-------------+-------|
| 충돌 | 교차점, 선·면, normal, feature, penetration, TOI를 계산하는 stateless query | tick 감시, broad-phase 후보, pair cache, contact 지속, event와 response |
| 운동 | kinematics, 힘, drag, bounce, spin과 ODE 공식 | body/environment/world state와 state 변환 |
| 궤적 | 보간, 미분, arc-length와 resampling kernel | sample 저장, impact metadata, 수명과 출력 |
| 제어·추정 | PID, Kalman, MPC 비용·최적화 kernel | sensor polling, target 선택, controller dispatch와 actuator 적용 |
| 변환 | vector/quaternion/transform 계산 | world bounds와 debug 출력 정책 |
| 상수 | 출처·version·단위가 있는 보편 상수 | 게임·제품 preset과 tuning 값 |

금지 의존성:

#+begin_src text
math-engine -X-> balix / entity / projectile / ground / navsys / GUI
navsys -X-> math-engine
balix / approved simulation consumers ---> math-engine
#+end_src

Navsys는 이 추출의 consumer가 아니다. 기존 Numal scalar 결합은 Navsys 선행
리팩터링에서 제거하며 Balix/Math Engine adapter를 Navsys에 추가하지 않는다.

Math Engine public header에는 =motion_state_t=, =environ_t=, =bodyprops_t=,
=trajectory_t=와 Balix callback type을 노출하지 않는다.

* P0 회귀 대상

1. =detect_triangle_collision_rotating_alpha=는 =theta=와 =tri_center=를 실제 회전에
   사용하지 않고 angle 0 placeholder를 반복한다.
2. moving triangle query는 상대 평면 root 뒤 world projectile point를 초기 triangle에
   검사하여 triangle translation을 inside test에 반영하지 않는다.
3. precise sphere Newton 경로는 bracket와 잔차 합격 없이 surface snap 후 success를
   반환할 수 있다.
4. trajectory 미분은 비균일 표본 공식을 사용하지 않고 resampling의 endpoint,
   누적오차와 duplicate-time 계약이 정의되지 않았다.
5. motion Verlet의 header/source const가 다르고, 이전 state를 덮은 뒤 angular 이전값을
   읽는다.
6. motion-model RK4가 초기화되지 않은 output acceleration을 먼저 scale한다.
7. transform의 non-uniform scale이 point apply, inverse와 composition translation에
   일관되게 적용되지 않는다.
8. ordinary quadratic은 header의 linear fallback 약속과 달리 =a ~= 0=에서 실패한다.
9. friction 모델의 coefficient 단위와 =v(1-mu*t)=, stop time =1/mu= 계약이 물리적
   마찰인지 감쇠인지 불명확하다.
10. controller PID factory는 전달받은 gain과 dt를 사용하지 않는다.
11. Kalman filter adapter가 서로 다른 callback signature를 function pointer cast로
    연결해 undefined behavior이고 measured velocity를 무시한다.
12. periodic environment 두 initializer가 계산에 사용되는 =delta_time=을 초기화하지
    않으며 callback 문서와 실제 signature도 다르다.
13. trajectory fresh init의 invalid delete, C ABI 밖 C++ allocation exception,
    truncation offset underflow와 =impact_time == 0= sentinel 충돌이 있다.
14. MPC quaternion angle이 normalize/clamp 없는 =acos(abs(w))=를 사용하고 route solver는
    진행 순서 대신 nearest point를 매번 다시 선택한다.
15. integrator default/clear는 environment method에 zero environment/body를 결합하고
    full init의 optional null field가 미초기화로 남을 수 있다.

기존 오류 동작을 golden answer로 승인하지 않는다. 분석적 해, 잔차, 보존량과
차원 검사를 기준으로 regression test를 작성한다.

Balix leaf 16개를 파일명 순서로 실행하지 않는다. 다음 wave를 따른다.

|------+-----------+-----------|
