* Aggregate direct export inventory

여기서 “직접 선언”은 해당 파일 본문에 실제로 적힌 =BYUL_API= 선언을 뜻한다. Include로
노출되는 child header의 symbol owner와 혼동하지 않는다.

| Header | 직접 =BYUL_API= 수 | 직접 export inventory | 단독 owner와 fate |
|--------+-------------------+-----------------------+-------------------|
| =byul/balix/balix.h= | 0 | 없음 | 새 symbol을 소유하지 않는 compatibility aggregate; child include와 link만 관리 |
| =byul/balix/numeq/numeq.h= | 0 | 없음 | 새 symbol을 소유하지 않는 deprecated compatibility aggregate; canonical internal include로 사용 금지 |
| =byul/balix/controller/controller.h= | 13 | controller adapter/lifecycle export 13개 | [[file:todo-balix-controller-adapter.org][Controller adapter 계획]]이 함수별 단독 소유; 이 문서는 aggregate/link 합류만 담당 |

=balix.cpp=의 =__balix_dummy__=와 =numeq.cpp=의 =__numeq_dummy__=는 =BYUL_API=가 아니며
public C ABI로 분류하지 않는다. Dummy translation unit의 존치 여부는 target이 실제
adapter source를 가진 뒤 결정한다.

* Controller adapter 합류 gate

Controller의 13개 =BYUL_API=, 2개 header inline, enum/callback/public layout, private
PID/Bang-Bang/MPC dispatch와 factory/allocator lifecycle은
[[file:todo-balix-controller-adapter.org][todo-balix-controller-adapter.org]]가 전수 소유한다.
이 문서는 다음 결과만 합류시킨다.

- controller adapter source는 Balix target에 남는다.
- PID/Bang-Bang/MPC 식과 optimizer는 Math Engine target에서만 compile된다.
- =controller.h=를 공개하는 installed target은 필요한 Math Engine public dependency를
  signature 노출 여부에 따라 =PUBLIC= 또는 구현 전용이면 =PRIVATE=로 정확히 설정한다.
- Root BYUL export의 controller 13개가 유지되고 canonical control symbol은 Math Engine
  manifest/DLL 한 곳에만 존재한다.

* 현재 CMake graph와 결함

현재 =byul/balix/CMakeLists.txt=는 다음을 동시에 수행한다.

- =balix_SOURCES=에 =${numal_SOURCES}=와 =${collision_SOURCES}=를 흡수한다.
- Body/environment, 모든 Numeq 구현, xform/trajectory/motion-state/controller source를
  하나의 =balix STATIC=에 직접 넣는다.
- =balix_HEADERS=에도 =${numal_HEADERS}=와 =${collision_HEADERS}=를 흡수한다.
- =balix STATIC=은 =core=와 =numal=을 link하고 =BYUL_STATIC=을 사용한다.
- =test_balix=는 =numal=, =balix=와 =entity=를 link하므로 독립 Math Engine consumer
  graph를 검증하지 않는다.

Root =byul/CMakeLists.txt=는 =numal_SOURCES=와 그 Numal source를 이미 포함한
=balix_SOURCES=를 다시 같은 =byul SHARED= inventory에 펼친다. 즉 Balix가 Math Engine
library를 소비하는 구조가 아니라 source를 흡수하며, source/header owner와 compile
mode가 중복된다.

* Target·DLL 이행 단계

| 단계 | Math Engine canonical source | Balix | Root =byul= shared | Gate |
|------+------------------------------+-------+--------------------+------|
| 현재 | 별도 target 없음 | Numal/collision/Numeq source 흡수 | =numal_SOURCES + balix_SOURCES= 재집계 | 중복 제거 필요 |
| Static 과도기 | 단일 manifest + =byul_math_engine_static= | source 흡수 제거, target link와 adapter만 소유 | 현 중앙 규칙에 따라 동일 manifest를 =BYUL_EXPORTS=로 직접 compile + legacy shim | module/root test와 source uniqueness |
| Shared 병행 | 같은 manifest로 =byul_math_engine= shared 검증 | imported target 소비 후보 | 기존 root 계산 export와 비교, 아직 cutover 전 | 중앙 규칙·runtime/package 승인 |
| 최종 cutover | Math Engine DLL이 canonical symbol 단독 소유 | imported shared target 소비 | canonical math source/symbol 제거, legacy shim만 유지 | =rules-ref.org= pin, export/import·two-DLL 검증 |

현재 중앙 규칙이 root shared 하나를 요구하므로 Static 과도기까지만 현 revision에서
진행할 수 있다. 독립 DLL 생성과 cutover는 중앙 Org 규칙 갱신 및 승인된
=rules-ref.org= revision 없이는 수행하지 않는다.

* Balix에 남는 책임

- world/entity/body/environment/controller state와 lifecycle
- tick attach/detach와 주기 실행
- broad-phase 후보, pair cache와 contact persistence
- collision event/callback, response와 impulse 적용
- trajectory sample/impact history와 serialization
- target, waypoint, sensor와 actuator policy
- 기존 public include/symbol의 compatibility forwarding

실제 projectile 주기 감시는 =projectile_tick.cpp=의 tick callback과 collision query 호출에
있다. Math Engine은 이 loop를 소유하지 않고 한 번의 query result만 반환한다.

* Adapter 원칙

- Balix DTO를 Math Engine POD/view로 명시 변환
- unit, world/local frame, time origin과 tolerance를 call site에서 지정
- =bool=만 전달하지 않고 status와 quality를 Balix event policy로 mapping
- Math Engine handle이 있으면 생성 library가 destroy
- callback, registry, tick과 mutable world pointer를 Math Engine에 넘기지 않음
- legacy wrapper 내부에서도 계산식을 다시 쓰지 않음
