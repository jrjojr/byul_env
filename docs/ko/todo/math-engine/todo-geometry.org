* 목적

이 문서는 BYUL Math Engine의 독립 =geometry= 계층 계획서이다. point, line, ray,
segment, primitive, curve, surface, mesh, NURBS, B-rep과 충돌·접촉 질의를 재사용
가능한 수학 API로 제공한다.

최종 목표는 게임 collision helper 하나가 아니라 계산기, 그래프·곡선 편집기, CAD,
게임, simulation, raycast/picking과 Python 도구에서 함께 사용하는 geometry kernel이다.

* 용어와 견고성 계약

이 문서의 상위 기능 영역은 =기하 계산(geometric computation)= 이다. 한국어 module
이름을 =강건=으로 축약하거나 이를 독립된 기하학 분야명처럼 사용하지 않는다.

=견고성(robustness)= 은 부동소수점 반올림 오차, 제한된 정밀도와 퇴화 입력에서도
기하 판정과 위상 관계가 서로 모순되지 않고 query가 선언된 상태로 종료하는 성질을
뜻한다. 제어 분야의 robust와 “불확실성 아래에서 요구 성질을 유지한다”는 상위 원리는
같지만, 여기서 다루는 불확실성은 plant·외란·sensor noise가 아니라 수치 오차와
기하학적 퇴화이며 유지 대상은 폐루프 안정성이 아니라 판정·위상 일관성이다.

견고성은 다음 개념과 같지 않다.

- accuracy: 계산값이 참값에 얼마나 가까운가
- exactness: 필요한 수치 또는 판정을 수학적으로 정확하게 계산하는가
- numerical stability: 계산 과정에서 오차가 과도하게 증폭되지 않는가
- robustness: 오차와 퇴화 입력에서도 결과 구조와 제어 흐름이 유효한가

정확 기하 판정, floating-point filter, interval error bound, 적응 정밀도와 명시적
degenerate 상태는 견고성을 달성하는 수단이다. 영문 API·설정 식별자에는 =robust=를
사용할 수 있지만 한국어 본문에서는 처음에 =견고성(robustness)=으로 정의하고 이후
=견고성= 또는 =오차에 견고한 기하 판정=으로 표기한다.

* 계층과 경계

#+begin_src text
numal
  scalar, vector, matrix, quaternion, interval, 오차에 견고한 기하 판정 기반
       ^
geometry
  primitive, curve, surface, mesh, NURBS, B-rep, query와 contact
       ^
plot adapter / CAD / game / balix adapter / Python
#+end_src

=geometry= 는 =numal= 에만 의존하는 것을 기본으로 한다. 고차 방정식과 최적화를 위해
=numeq= 를 선택 private dependency로 사용할 수 있지만 public 자료형에 노출하지 않는다.

=geometry= 는 =balix=, entity, projectile, ground, GUI와 renderer를 include하지 않는다.
점·선·면의 교차 좌표, swept query, TOI, 접촉점과 manifold처럼 snapshot 입력으로
계산하는 stateless geometry/CCD는 시간·속도를 사용해도 Math Engine 책임이다. Balix는
tick 감시, world lookup, broad-phase 후보, contact persistence, event와 response를
소유한다. Impulse/constraint 공식은 Math Engine physics가 제공할 수 있지만 어느
object에 언제 적용할지는 Balix 책임이다.

* Backend 기본 결정

BYUL은 임베디드와 실시간 사용을 위한 자체 =geometry_core= 를 유지하고, 고급 CAD
기능은 Open CASCADE Technology(OCCT)를 선택 backend로 검토한다.

#+begin_src text
geometry_core
  primitive, closest point, raycast, SAT/GJK, contact
  Bezier와 기본 spline
  embedded full/compact/minimal 공통 기반
       ^
geometry_occt
  NURBS surface, trimmed surface, B-rep, healing, boolean, STEP/IGES
  desktop/full/CAD profile 전용 선택 backend
#+end_src

FreeCAD의 Part/CAD 모듈을 BYUL library 계층으로 직접 편입하지 않는다. FreeCAD는
application/document/workbench/Python/Qt 계층과 결합되어 있으므로 사용 사례, UI와
test를 참고하되 CAD kernel은 필요하면 OCCT를 직접 연결한다.

OCCT와 FreeCAD source의 일부를 BYUL source에 복사하지 않는다. 외부 backend는
private C++ adapter 뒤에 숨기고 public C ABI에는 OCCT의 =TopoDS_Shape=,
=Handle(Geom_*)= 또는 기타 외부 타입을 노출하지 않는다.

* 공통 수치·결과 정책

- 전역 epsilon 하나를 사용하지 않음
- absolute, relative, angular와 parameter tolerance 분리
- 좌표 scale과 bounding extent 고려
- degenerate, tangent, coincident와 overlap 상태 구분
- 정확/적응형 기하 판정과 floating-point filter 후보
- =f32= 실시간, =f64= 기본 engineering backend
- =f128=/arbitrary precision 검증 backend 후보
- NaN, infinity, overflow, max iteration과 workspace 상태
- 실패 시 입력과 출력의 부분 변경 금지

mpdecimal은 geometry kernel의 기본 좌표 backend로 사용하지 않는다. CAD 치수 입력,
단위 변환, 공차 설정 UI, 제조 치수 직렬화와 결과 보고에는 =mathunits/numal decimal=
adapter를 사용할 수 있지만 raycast, GJK/EPA, mesh/NURBS tessellation과 contact 반복은
=f64= 및 별도 견고한 기하 판정 backend를 기본으로 한다.

geometry API도 게임용과 공학/CAD용으로 복제하지 않는다.

- speed preset: f32 후보, 제한 반복, 단순 primitive, SAT/GJK fast path
- precision preset: f64 이상, 견고한 기하 판정, refinement, 엄격 tolerance와 오차 보고
- 순수 bounding-volume overlap query는 speed, 애매한 primitive intersection과 접촉
  시점은 precision으로 승격 가능. Broad-phase 후보 생성·scheduling은 Balix/caller 책임
- 승격 전후 query result의 feature id와 의미 계약 유지

공통 policy는 =todo-math-computation-context.org= 를 따른다.

모든 query는 bool 하나가 아니라 상태와 품질을 반환한다.

parallel / coincident / degenerate / non-finite
max-iterations / insufficient-workspace / unsupported
#+end_src

** ABI와 ownership

- OCCT object는 =byul_geometry_occt= 내부 opaque handle로만 소유
- handle을 생성한 backend/DLL이 동일 handle을 destroy
- OCCT allocator, RTTI와 exception이 C ABI 밖으로 나가지 않음
- OCCT version과 compiler ABI가 BYUL public structure layout을 바꾸지 않음
- backend가 없는 환경에서는 stable unsupported 상태 반환

** 라이선스와 배포 조건

OCCT는 사용 version의 LGPL 2.1과 OCCT exception 원문을 기준으로 검토한다. 상용
license 또는 별도 계약을 선택한 경우 해당 계약을 별도로 기록한다.

- BYUL과 OCCT binary/source의 license 범위를 분리
- BYUL의 비상업·재배포 제한을 OCCT에 적용하지 않음
- 사용자에게 OCCT 사용 사실과 권리를 명확히 고지
- LGPL 2.1, OCCT exception과 copyright notice를 package에 포함
- 사용한 정확한 OCCT version, build option과 patch 기록
- 배포 binary에 대응하는 OCCT source 또는 동일 위치의 source 접근 수단 제공
- 수정한 OCCT를 배포하면 해당 수정 source와 build material 제공
- shared library 교체를 막는 기술적·license 제한을 두지 않음
- static link는 relink 가능한 object 제공 의무를 검토하고 기본 배포에서는 피함
- =THIRD_PARTY_NOTICES.md= 와 =licenses/= package layout 추가

권장 기본은 별도 OCCT shared library와 동적 연결이다. BYUL 문서와 package license에
third-party component는 각자의 license만 적용된다는 carve-out을 둔다. 실제 배포
전에는 선택 version과 패키징 형태를 기준으로 법률 검토한다.

* 완료 기준

- 독립 =geometry= target이 상위 simulation/UI에 의존하지 않는다.
- primitive의 거리·교차·최근접점 API가 일관된 result를 제공한다.
- collision result가 위치, parameter, normal, distance, penetration과 feature id를 가진다.
- 여러 접촉점을 manifold로 표현한다.
- Bezier, spline와 NURBS curve/surface의 평가·미분·분할·tessellation을 지원한다.
- curve-line, curve-curve와 curve-surface intersection이 parameter와 오차를 보고한다.
- B-rep geometry/topology가 분리되고 validation 가능한 handle을 가진다.
- 동적 collision의 stateless CCD/TOI 계산은 Math Engine, 감시·상태·event는 Balix
  adapter로 분리된다.
- OCCT가 없어도 core SDK와 embedded profile이 build·test된다.
- OCCT backend는 public ABI를 오염시키지 않고 shared library로 교체 가능하다.
- third-party license와 BYUL license 범위가 package에서 명확히 분리된다.
- embedded profile별 기능과 workspace 상한이 정의된다.
- C API, expression/plot adapter와 Python wrapper가 동일 kernel을 사용한다.
