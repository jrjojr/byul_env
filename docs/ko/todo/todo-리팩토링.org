* 목적

BYUL 저장소의 모든 =.h=, =.hpp= 파일을 중앙 =rules_env=의 선언, 주석, 작성,
public ABI, build와 test 규칙에 맞게 단계적으로 리팩터링한다. 이 계획은 public header만
다루지 않는다. SDK public, compatibility, internal/detail, generated, tool-only,
reference/deprecated header를 모두 inventory에 넣고 각 파일의 역할과 소비자를 먼저
확정한다.

헤더 문구를 일괄 치환하는 작업이 아니다. 선언과 구현, module static target, root shared
library, 설치 SDK, wrapper, 외부 C consumer가 같은 계약을 검증한 뒤에만 한 단계를 닫는다.

* Static과 shared의 불변식

BYUL native build의 기준 관계는 다음과 같다.

#+begin_src text
module source/header inventory
  +-- module static target: 개발과 module test, BYUL_STATIC
  +-- root shared target: 같은 inventory를 다시 compile, BYUL_EXPORTS
                         Windows byul.dll / Unix libbyul.so
#+end_src

최종 runtime shared library는 module static archive를 평범하게 link하여 만들지 않는다.
그 방식은 참조되지 않은 public object와 symbol을 linker가 제거할 수 있기 때문이다.
object library 또는 whole-archive 방식으로 변경하려면 export, PIC, Windows/Linux/macOS
symbol 검증을 먼저 추가한다. module static test 통과만으로 root shared 공개 API가
완성됐다고 판정하지 않는다.

Math Engine 분리는 두 phase로 실행한다. 중앙 규칙 변경 전에는 계산 source의 canonical
소유권과 component static target을 분리하되 현행 root shared가 같은 inventory를 다시
compile한다. [[file:math-engine/todo-math-engine-build-abi.org][Math Engine Build·DLL ABI TODO]]의
규칙 변경과 =rules-ref.org= 갱신 gate를 통과한 뒤에는 canonical math symbol을 독립 Math
Engine DLL 한 곳만 export하고 root/Balix는 link와 legacy forwarding만 수행한다.

* 목표 Header 상태

각 header는 machine-readable role manifest에서 정확히 하나의 primary role을 가진다.

#+begin_src text
public-foundation | public-module-entry | public-aggregate | public-component
compatibility-forwarder | cpp-facade
internal | generated | tool-only | reference | deprecated-reference
#+end_src

각 최초 header asset은 primary role, 현행 경로와 최종 disposition을 가진다. 단순
rename이면 하나의 canonical 경로를 가질 수 있지만, Balix → Math Engine 추출처럼 책임을
분리하는 작업은 기존 compatibility/adapter header와 0개 이상의 새 canonical successor
header를 가질 수 있다. 최초 header와 successor header를 일대일 rename으로 가정하지 않는다.
리팩터링 완료 시점에는 이름을 그대로 유지한 header도 명명 규칙에 맞는다는 판정 근거가
있어야 하며, 이름이 규칙과 맞지 않는 header는 구현·include·install·wrapper·문서와 함께
일관되게 전환한다.

** Header 파일명 불변식

- 대표 public type을 직접 제공하는 canonical header는 대표 type stem을 사용한다.
- 대표 type이 없는 public-component는 중앙 header 규칙의
  =<public_module>_<responsibility>.h= 형식을 사용한다.
- Project명, 상위 분류 directory와 implementation directory chain을 basename에
  기계적으로 이어 붙이지 않는다. Public module 경계는 role manifest에 명시한다.
- SDK에 설치되는 모든 public header basename은 전역에서 유일해야 한다.
- Internal/detail header는 SDK와 wrapper 입력에서 제외하고 소유 module 내부에서
  responsibility가 드러나는 이름을 사용한다. 물리 directory 전체를 basename으로
  복제하지 않는다.
- Generated header는 산출물만 rename하지 않고 generator/template과 canonical output
  path를 함께 변경한다.
- Tool-only header는 해당 tool의 module 경계를 사용하며 BYUL SDK public module처럼
  가장하지 않는다.
- Compatibility-forwarder만 legacy basename을 예외로 유지할 수 있다. 이 경우
  canonical target, deprecated 상태와 제거 예정 version을 manifest에 기록한다.
- 하나의 기존 header가 여러 책임을 포함하면 =split= disposition을 사용하고
  =compatibility_path=, =successor_headers[]=, 각 successor의 =canonical_owner=와
  =design_todo=를 기록한다. 기존 header 자체의 이름과 새 header 이름을 하나의
  =canonical_path= 값으로 덮어쓰지 않는다.

모든 rename은 파일만 이동하는 작업이 아니다. =@file=, include guard, source/test include,
module/root CMake inventory, install/package, aggregate header, wrapper manifest, 생성물과
공개 문서를 같은 단계에서 갱신한다. Public header rename은 source compatibility,
exported symbol rename은 ABI compatibility 변경으로 별도 판정한다.

public C header의 완료 상태:

- 표준 license가 첫 block이며 정확한 =@file=, =@brief=와 역할 설명이 뒤따른다.
- =BYUL_= guard, direct standard/project include, include 순서, C linkage가 독립적이다.
- exported function은 =BYUL_API=와 C ABI를 사용하며 C17/C++17에서 단독 include된다.
- pointer direction, nullability, ownership/lifetime, invalidation, thread/reentrancy, error,
  unit/range/nonfinite, buffer capacity와 callback userdata 계약이 =@byul.*= metadata와
  Doxygen으로 표현된다.
- value/resource/enum ABI는 size, align, offset, numeric value, support 범위와 fingerprint로
  동결된다.
- header, implementation signature, root export, wrapper와 installed SDK가 같은 inventory를
  사용한다.
