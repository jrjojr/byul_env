** Symbol disposition 문법

| 원자 token | 의미 |
|------------+------|
| =keep= | 현행 symbol과 signature/numeric value/layout을 해당 ABI 단계에서 유지 |
| =add= | 기존 ABI를 건드리지 않고 새 canonical symbol을 추가 |
| =deprecate-forward= | legacy symbol을 보존하고 canonical implementation으로 forwarding |
| =internalize= | SDK/install/wrapper에서 제거하고 구현 전용으로 이동; 이미 공개된 symbol이면 ABI-major에서만 실행 |
| =abi-major= | signature, layout, enum 숫자 또는 제거처럼 ABI-major가 필요한 직접 변경 |
| =conformance-block= | legacy export는 유지하지만 correctness gate 전에는 새 facade/capability의 지원 경로로 승격하지 않음 |
| =decision-required(A|B)= | 아직 결론이 나지 않은 선택; child를 =DONE=으로 바꾸기 전에 반드시 하나로 축약 |

하나의 현행 symbol은 단일 token만 억지로 붙이지 않고 시점별 chain을 쓴다. 예를 들어
=ABI 1.x: deprecate-forward -> ABI-major: internalize=는 허용한다. 기존 symbol 유지와 새
companion 추가가 동시에 필요하면 =ABI 1.x: keep; companion: add=처럼 대상을 분리한다.
=keep+add=, =keep/deprecate-forward=, =deprecate=, =block-forward until corrected= 같은 축약은
각각 위 정규형으로 풀어 쓴다. =conformance-block=은 최종 제거 처분이 아니므로 반드시 해당
ABI 단계의 =keep= 또는 =deprecate-forward=와 함께 기록한다.

* Numal/Math Engine 분리 불변식

Navsys는 Numal 또는 Math Engine의 consumer가 아니다. Grid/path/maze/search와 D* Lite의
수치 비교, 각도 표시와 value copy 정책은 Navsys가 소유하며 다음을 만족해야 한다.

- Public Navsys header는 =scalar.h=, Numal 또는 Math Engine header를 include하지 않는다.
- =dfs.h=와 =weighted_astar.h=의 사용하지 않는 =scalar.h= include를 제거한다.
- D* Lite key/order와 g/rhs 비교는 Navsys가 명시한 exact/tolerance policy를 사용하고
  범용 =scalar_equal=에 의존하지 않는다.
- =route.cpp=의 degree 변환은 implementation-local C++17 상수와 =<cmath>=를 사용하며
  =M_PI= 또는 Math Engine unit API를 요구하지 않는다.
- =scalar_copy/scalar_destroy=는 Math Engine 함수가 아니라 현재 =coord_hash=가 소유한
  legacy helper다. 대표 type/module prefix로 forwarding하거나 internal callback으로
  이동한다.
- Navsys와 하위 module/test CMake에서 Numal include directory와 =numal= link dependency를
  제거한다.
- Math Engine을 구현한 뒤에도 이 dependency를 다시 추가하지 않는다.

허용되는 외부 기반은 BYUL config/core ABI, 명시된 =byul_tick= lifecycle과 C/C++17 표준
library다. 향후 실제 범용 Math Engine 계산이 필요하면 이 TODO의 범위를 암묵적으로
확장하지 않고 별도 승인된 dependency 변경으로 다룬다.

* 의존성 순서

#+begin_example

  -> route value + navcell -> navgrid
  -> obstacle core -> obstacle generator -> route_carver
  -> maze core/options -> 각 generator -> maze dispatcher
  -> route-finder evaluation/binding -> 각 algorithm -> route-finder dispatcher

dependency 순서를 건너뛰지 않는다.

한 문서가 다른 child 소유 source까지 수정하도록 적혀 있으면 해당 문구를 consumer 합류 또는
최종 통합 검증으로 바꾼 뒤 착수한다. 같은 source, test 또는 public symbol을 두 child에서
동시에 =DOING=으로 두지 않는다.

* Build와 배포 불변식

[[file:../../../../byul/navsys/CMakeLists.txt][navsys CMake]]는 module static target을 만들고
=BYUL_STATIC=으로 컴파일한다. [[file:../../../../byul/CMakeLists.txt][root byul CMake]]는 module
library를 그대로 링크하는 모델이 아니라 =navsys_SOURCES=를 root shared source에 다시 넣고
=BYUL_EXPORTS=로 재컴파일한다. 따라서 다음 네 결과는 서로 대체할 수 없다.

- module static build/test 성공
- root shared에서 동일 source 재컴파일과 export table 성공
- build-tree와 install-tree의 C/C++ direct-header consumer 성공
- wrapper manifest/parser/generation/load와 runtime ownership test 성공

header 하나를 완료하려면 해당 child가 네 축 중 영향을 받는 모든 축의 검증 명령과 expected
result를 가져야 한다. C++ helper를 SDK에서 제외하는 일도 module compile 성공만으로 완료하지
않고 install manifest와 기존 consumer migration까지 검증한다.

* Dry-run 기준선과 단계별 fail-closed gate

리팩토링 전 dry-run은 구현 완료 판정이 아니라 현재 결함과 이후 regression을 분리하는
machine-readable 기준선이다. 다음 결과를 서로 독립적으로 보존한다.

- CFFI generated block의 =--check= 결과와 등록 header 집합
- Navsys 41개 public C header 및 2개 C++ helper의 wrapper 지원 처분
- header declaration, CFFI declaration, handwritten =C.*= 참조와 root shared export의
  교집합 및 차집합
- header audit issue를 header, symbol과 diagnostic code별로 정규화한 snapshot
- module static, root shared, Python/native integration의 test case와 assertion 결과

Wrapper symbol 검사는 모든 concrete wrapper module을 먼저 import하여 공유 =ffi=에 각
=cdef= block을 등록한 뒤 수행한다. 등록 전 =ffi_core.C=만 조회한 결과는 유효한 symbol
검사가 아니다. 미해석 참조는 다음 두 종류를 구분하며 둘 다 touched surface의 완료를
막는다.

| 분류 | 의미 | 완료 조건 |
|------+------|-----------|
| missing-cdef | handwritten wrapper가 manifest/header에 없는 이름을 참조 | canonical symbol로 전환하거나 명시적으로 wrapper method 제거 |
| missing-export | CFFI 선언은 있으나 root shared에 symbol이 없음 | root source/export 누락 수정 또는 ABI 호환 처분 확정 |

Header audit는 기존 부채 때문에 전체 migration을 한 번에 막지 않도록 다음 세 gate로
운영하되 최종 기준을 낮추지 않는다.

| gate | 적용 시점 | 통과 조건 |
|------+-----------+-----------|
| migration | child 착수 전후 | 기존 issue가 증가하지 않고 새 diagnostic code가 생기지 않음 |
| touched-header | 각 child 완료 | 수정한 public header의 audit error가 0 |
| Navsys release | 43개 통합 | Navsys public header audit error 0, 미등록·무처분 header 0 |

Repository 전체 =--audit --all-headers= 오류는 BYUL 전체 release gate가 소유한다. Navsys
밖의 기존 오류를 Navsys child의 구현 실패로 섞지는 않지만, 전체 release 성공으로도
보고하지 않는다.

각 child는 구현 직후 다음 local integration loop를 닫고 다음 dependency로 이동한다.

#+begin_example
header audit/check
  -> module static build/test
  -> root shared 재컴파일/export resolution
  -> affected wrapper symbol resolution
  -> Python result/status/ownership/lifetime test
#+end_example

마지막 통합 단계는 이 검사를 처음 수행하는 단계가 아니라 모든 child에서 통과한 증거를
clean build/install 환경에서 반복하는 release gate다. Class 이름이 test source에 한 번
등장하거나 route를 출력만 하는 test는 wrapper 동작 coverage로 인정하지 않는다.
