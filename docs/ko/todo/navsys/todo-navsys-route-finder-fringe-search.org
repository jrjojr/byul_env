* 대상과 원전 기준

대상은 =fringe_search.h/fringe_search.cpp=다. 공통 API는
[[file:todo-navsys-industry-api-baseline.org][Navsys 업계 API 기준]]에 연결한다.
Algorithm conformance는 Björnsson/Enzenberger/Holte/Schaeffer의
[[https://staff.ru.is/yngvi/w_publications.html][Fringe Search 원 저자 publication]] 및 공개 논문
“Fringe Search: Beating A* at Pathfinding on Game Maps”의 f-limit/f-min, two-list와 cache
locality 모델을 기준으로 한다.

* 현재 구현/소비 분석

- =find_fringe_search= 하나가 module/root/SDK/generated CFFI/dispatcher에 공개된다.
- Header는 two priority queue라고 설명하지만 원 알고리즘의 핵심은 정렬 OPEN을 제거한
  list traversal이다. 구현도 =cost_coord_pq_t= 두 개를 사용해 “Fringe Search” 성능/복잡도
  주장을 그대로 적용할 수 없다.
- =delta_epsilon=과 경험적 10x10 권장값은 원전의 stable contract가 아니며 구현은 nonpositive를
  0.5로 silent 교정한다. Dispatcher/navsys helper 기본은 각각 0.3으로 또 다르다.
- 내부 =expanded=는 이 round에서 relax가 하나라도 있었는지만 나타낸다. Deferred frontier가
  남아 있어도 =!expanded=면 종료하여 completeness를 해칠 수 있다.
- callback userdata가 없고 항상 NULL, =visited= hash는 결과 판단 없이 memory만 소비한다.
  Nonpositive max_retry는 unlimited인 반면 다른 algorithm은 invalid다.

옮긴다.
