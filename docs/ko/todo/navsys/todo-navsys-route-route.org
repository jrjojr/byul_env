* 문서 책임

대상은 =byul/navsys/route/route.h=와 구현 =route.cpp=다. 이 문서는 route result value,
route builder, direction과 heading tracker의 선언·구현·ABI 및 Python =route.py= value
wrapper만 소유한다. 공통 status, allocator, ownership, two-call buffer,
ABI/static/shared/wrapper gate는
[[file:todo-navsys-industry-api-baseline.org][Navsys 업계 API 기준]]을 따른다.

이 문서는 route를 단순 좌표 list가 아니라 search 결과 value로 취급한다. 좌표열,
완료 상태, cost, 통계, trace의 불변식을 먼저 정하고, 경로 편집은 별도 builder,
시간에 따라 누적되는 heading 판정은 별도 state object로 분리한다.

* 실행 소유권과 Consumer 합류 Gate

- 이 문서는 route value/builder와 그 단위 test만 수정한다. 각 search algorithm의
  route 생성 절차, finder adapter, finder config/callback과 =route_finder.py= 전환은
  소유하지 않는다.
- Route canonical API가 준비되면
  [[file:todo-리팩토링-navsys-headers.org][Navsys Header 오케스트레이터]]의 root 통합에서
  route-finder와 =navsys.cpp= consumer를 한 번에 전환한다. Consumer 전환을 이 문서와
  개별 finder 문서에서 중복 실행하지 않는다.
- Route 단위 assertion의 실제 파일은
  =byul/navsys/route/tests/test_module_route.cpp=다.
- Finder consumer 전환 assertion은 root 통합이
  =byul/navsys/route_finder/tests/test_module_route_finder.cpp=,
  =byul/navsys/route_finder/tests/test_module_route_finder_discrete.cpp=와
  =byul/navsys/tests/test_module_navsys.cpp=에 추가한다.
- Python route value/lifetime assertion은
  =tools/python/byul_wrapper/tests/test_route.py=에 추가하고, finder wrapper consumer
  assertion은 root 통합이
  =tools/python/byul_wrapper/tests/test_route_finder.py=에 추가한다.

* 조사한 현재 소비 경로

- =byul/navsys/route/CMakeLists.txt=와 =byul/navsys/CMakeLists.txt=가 module static에,
  =byul/CMakeLists.txt=가 같은 =route.cpp=를 root shared에 직접 compile하며
  =route.h=를 설치 SDK에 배포한다.
- A*/BFS/DFS/Dijkstra 및 나머지 route-finder 구현이 =route_create=,
  =route_add_coord=, =route_add_visited=, public =cost/success= field와
  =route_reconstruct=를 혼합해 결과를 만든다.
- =tools/python/byul_wrapper/byul_wrapper/route.py=는 struct layout과 함수 선언을
  CFFI에 등록한다. =coords/visited_order/visited_count= wrapper는 borrowed child인데
  parent route를 강하게 보관하지 않는다.
- Python =make_direction()=과 =direction_to_coord()=는 native가 새로 할당한
  =coord_t*=를 =own=False=로 감싸므로 정상 호출마다 누수된다. wrapper는 header에 없는
  =route_reconstruct_path=도 호출한다.
- 현재 test는 append/slice/direction의 일부 happy path만 다루고, OOM, NULL,
  invalid index, metadata 불변식, borrowed lifetime, reconstruction cycle과 wrapper leak을
  검증하지 않는다.
- =route.cpp=는 degree 변환을 위해 Numal =scalar.h=의 =M_PI= macro를 사용한다. 이는
  Route API가 Math Engine 계산을 요구하는 것이 아니므로 implementation-local C++17
  수학 상수와 =<cmath>=로 교체한다.

* 업계 기준과 직접 비교

- [[https://recastnav.com/classdtNavMeshQuery.html][Detour dtNavMeshQuery]]의 =findPath=와
  =findStraightPath=는 caller가 제공한 buffer, 실제 count와 상세 status를 함께 사용하며
  partial result를 성공/실패 NULL 하나로 숨기지 않는다. route 좌표 export도 이 형태를
  기본 C ABI로 삼는다.
- [[https://recastnav.com/DetourStatus_8h_source.html][Detour status]]는 success,
  in-progress, partial, invalid parameter, out-of-memory, buffer-too-small과 같은 원인을
  bit/detail로 구분한다. =success bool + NULL + 0 sentinel= 혼합을 교체할 기준이다.
- [[https://www.boost.org/doc/libs/latest/libs/graph/doc/astar_search.html][Boost.Graph A*]]는
  predecessor/distance map과 traversal event를 결과 path 자체와 분리한다. 따라서
  =visited_order/visited_count/total_retry_count=는 route value가 아니라 search
  stats/trace의 책임으로 이동해야 한다.
- [[https://curl.se/libcurl/c/][libcurl C API]]의 explicit handle lifecycle처럼 create/destroy,
  borrowed view와 caller-owned output의 수명을 함수명·문서·wrapper에 동일하게 드러낸다.
- [[https://isocpp.github.io/CppCoreGuidelines/CppCoreGuidelines#i11-never-transfer-ownership-by-a-raw-pointer-t-or-return][C++ Core Guidelines I.11]]의 ownership 원칙은 C ABI에서도 allocation을 숨기는
  =make/get/to_coord= 이름을 피하고 =create/copy/destroy= 또는 caller output으로 소유권을
  노출해야 함을 뒷받침한다.

* 확인된 P0/P1 결함과 의미 충돌

1. public =s_route=가 세 owning container pointer와 mutable 결과/heading state를 노출한다.
   어느 field라도 외부가 교체하면 destructor double-free/leak 및 metadata 불일치가 난다.
2. =route_hash=와 =route_equal=은 pointer identity인데 =route_copy=는 deep copy다. 이름만 보면
   value hash/equality로 오해하며 copied route는 같은 내용이어도 equal하지 않다.
3. =route_get_total_retry_count=와 =route_set_total_retry_count=는 NULL을 바로 역참조한다.
   다른 getter의 NULL sentinel 정책과도 다르다.
4. =route_get_cost(NULL)=0, =route_get_success(NULL)=0, =route_length(NULL)=0은 유효한
   zero-cost/unsuccessful/empty route와 invalid handle을 구분하지 못한다.
5. append/insert/remove/clear는 좌표만 바꾸고 cost, success, visited data, heading accumulator를
   재계산하거나 invalidate하지 않는다. =route_append=는 두 route의 metadata를 병합하지 않고,
   =route_append_nodup=는 접합점 하나만 중복 제거한다.
6. =route_slice=는 coords만 복사하고 cost/success/visited/retry/heading을 기본값으로 잃는다.
   범위는 half-open인지 문서화되지 않고 =end == start= empty slice도 거부한다.
7. void mutator는 invalid index와 allocation failure를 조용히 무시하여 strong guarantee와
   호출자 복구 경로가 없다. =route_add_visited=는 order와 count의 두 단계 변경을 원자적으로
   보장하지 않는다.
8. =route_make_direction=과 =direction_to_coord=는 조회처럼 보이지만 heap allocation한다.
   invalid 입력도 allocated =(0,0)=을 반환해 invalid와 unknown을 합치고 wrapper가 이를 샌다.
9. =route_get_direction_by_dir_coord=, =calc_direction=과 방향 변화 API는 NULL coord를
   역참조한다. =calc_direction=과 =direction_to_coord=는 =route_= prefix도 없다.
10. =route_has_changed_with_angle=은 predicate 이름인데 =avg_vec_x/y/vec_count=를 갱신한다.
    =route_has_changed=는 누적 vector를 정규화하지 않아 같은 방향을 여러 번 누적하면 dot가
    1로 clamp되어 실제 각도를 왜곡한다. 두 함수의 state transition도 서로 다르다.
11. =route_reconstruct=는 predecessor cycle을 탐지하지 않아 무한 loop가 가능하고,
    temporary/append allocation failure를 검사하지 않는다. 실패 시 기존 route에 일부 좌표가
    남을 수 있으며 algorithm-internal =coord_hash_t= representation을 public ABI로 누출한다.
12. enum의 암시적 숫자, generic =ROUTE_H= guard, license/header comment 누락,
    =route_create()=의 C prototype 부재, C header에서 =bool/uintptr_t=를 직접 사용하면서
    required standard include가 보이지 않는 문제가 header 규칙과 충돌한다.

* ABI, static/shared와 Wrapper 전환

- ABI 1.x에서는 enum 숫자, =s_route= size/align/field offset과 44개 symbol/signature를
  manifest로 동결한다. crash/누수는 동작 호환 범위 안에서 고치고 canonical API를 additive
  export한다.
- 이 문서는 builder/internal result writer와 public accessor를 제공한다. Repository
  algorithm의 소비 전환과 public field direct access 제거는 root 통합이 수행하며, 그
  결과가 0건인 것을 ABI 2 전환 Gate에서 확인한다.
- module static target와 root shared target가 같은 implementation/header inventory 및 compile
  definition을 사용하고, legacy/canonical symbol이 shared DLL에 정확히 한 번 export되는지
  확인한다.
- CFFI는 ABI 1 struct layout을 generated fingerprint와 함께 유지하되 새 wrapper는 opaque
  canonical API만 사용한다. borrowed child wrapper는 parent를 강하게 보관하거나 아예 value
  copy fetch로 교체한다.
- allocation을 제거하기 전까지 legacy =route_make_direction/direction_to_coord= 반환은
  Python에서 반드시 =own=True=로 감싸고, wrapper의 존재하지 않는
  =route_reconstruct_path= 호출은 제거한다. replacement는 C header에서 생성된 declaration을
  단일 source of truth로 등록한다.
