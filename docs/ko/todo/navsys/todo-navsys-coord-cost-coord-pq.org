* 현재 구현 근거

- storage는 =std::map<float, std::vector<coord_t*>>=이며 push마다 coordinate를 heap copy한다.
- 같은 cost는 vector =back()=을 사용하므로 LIFO다. tie 정책이 문서/테스트에 없다.
- =float= NaN이 strict weak ordering을 깨뜨릴 수 있고 Inf/음수 허용 여부도 없다.
- =peek=은 mutable borrowed pointer, =pop=은 owned pointer를 반환하지만 이름과 문서가
  소유권 차이를 드러내지 않는다.
- Python wrapper는 =pop= 결과를 non-owning으로 감싸므로 반환 allocation을 leak한다.
- empty =peek_cost=가 0을 반환해 valid zero cost와 구별되지 않는다.
- =remove=는 float exact equality bucket에서 같은 coordinate를 모두 제거할 수 있다.

* 업계 비교

- 표준 priority queue와 [[https://www.boost.org/doc/libs/latest/doc/html/heap/concepts.html][Boost.Heap 공식 interface]]는 top key/value와 update/handle 정책,
  comparator 전제와 invalidation을 분명히 한다. Mutable heap은 handle로 값을 바꾼 뒤
  heap order를 복구하는 update/increase/decrease를 구분한다. Navsys public C API는 구현
  container를 노출하지 않고 caller-storage status operation을 제공한다.
- 비교 key에 NaN을 허용하지 않거나 canonical total-order를 별도 정의해야 한다. 경로
  비용 queue의 기본은 finite non-negative cost를 validation하는 것이다.
- deterministic navigation test를 위해 equal-cost tie는 FIFO insertion sequence 또는
  좌표 lexicographic 중 하나를 public 계약으로 고정한다. 권장은 stable FIFO다.

** ABI 1.x migration table

| 현행 선언 | 관찰된 ABI 1.x 계약 | canonical 이행 |
|-----------+---------------------+----------------|
| =cost_coord_pq_t= | opaque queue resource | keep |
| =cost_coord_pq_create= | allocation 실패 채널이 없는 caller-owned queue | =create_ex=로 deprecate-forward |
| =cost_coord_pq_destroy= | NULL 허용, queue와 남은 coordinate copy 파괴 | keep |
| =cost_coord_pq_push= | NULL no-op, coordinate heap copy, 모든 float 허용 | validation하는 =push_ex=로 deprecate-forward |
| =cost_coord_pq_peek= | 최소 cost bucket의 마지막 element를 mutable borrowed 반환 | =peek_min=으로 deprecate-forward |
| =cost_coord_pq_pop= | =peek=과 같은 pointer를 제거해 caller-owned 반환 | =pop_min=으로 deprecate-forward |
| =cost_coord_pq_peek_cost= | NULL/empty와 valid zero가 모두 =0.0f= | =peek_min=으로 deprecate-forward |
| =cost_coord_pq_is_empty= | NULL도 empty, bucket을 순회 | const =empty=로 keep/forward |
| =cost_coord_pq_contains= | NULL false, cost 무관 coordinate 선형 검색 | const receiver로 keep/forward |
| =cost_coord_pq_remove= | =-0/+0= 동일 bucket, exact-cost의 같은 coordinate 전부 제거 | =remove_one/remove_all=로 deprecate-forward |
| =cost_coord_pq_length= | NULL 0, 전체 bucket O(n) 합산과 int truncation | O(1) =size=로 deprecate-forward |
| =cost_coord_pq_trim_worst= | =n <= 0= no-op, 큰 cost부터 정확히 =n=개 제거 | =trim_to_size=로 deprecate-forward |

ABI 1.x 동률 순서는 동일 cost vector의 =back()=을 사용하는 LIFO로 보존한다. Canonical
API는 deterministic navigation을 위해 insertion sequence 기반 FIFO를 사용한다.
=peek= pointer는 다음 mutation 또는 destroy 전까지만 유효하다. 동일 항목을 =pop=하면
그 pointer 자체가 caller-owned로 이전되며 caller는 정확히 한 번 =coord_destroy=해야 한다.

* 목표 의미

** key domain과 ordering

- =isfinite(cost) && cost >= 0=을 기본 precondition으로 한다. 음수 비용이 실제 필요한
  consumer가 확인되면 create-info flag로 허용하되 NaN은 계속 거부한다.
- equal cost tie는 monotonically increasing insertion sequence의 FIFO로 한다. sequence
  overflow 시 재번호화/오류 정책을 test한다.
- =-0.0f=는 =+0.0f=로 canonicalize한다. remove는 입력 cost를 exact canonical key로
  사용할지 coordinate-only removal을 canonical로 할지 분리한다.

** storage와 ownership

- entry는 ={float cost; coord_t coord; uint64_t sequence}=를 by-value 저장한다.
- peek/pop은 caller가 제공한 =coord_t=와 float에 복사한다. pop 성공 직후에도 별도
  destroy가 필요하지 않다.
- 내부 pointer를 유지해야 하는 ABI 1 =peek=은 다음 mutation/destroy 전까지 borrowed,
  =pop=은 =coord_destroy= 필요라는 legacy 문서를 정확히 보강한다.

** mutation과 complexity

- =remove_one=과 =remove_all=을 구분하고 removed count를 반환한다.
- =size=는 O(1) entry count를 유지하며 =size_t=를 사용한다.
- =trim_worst(max_size)=는 최대 size로 자르고 제거 수를 반환한다. 기존 parameter =n=이
  keep-count인지 remove-count인지 characterization 후 adapter semantics를 보존한다.
