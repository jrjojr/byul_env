* 현재 구현 근거

- storage는 =std::vector<coord_t>=이고 list object만 opaque다.
- =get/front/back=은 내부 element의 borrowed pointer를 반환한다. push/insert/remove/
  reverse/clear와 vector reallocation 뒤 모두 invalid될 수 있지만 문서에 없다.
- 생성/복사/mutation에서 C++ allocation exception이 C ABI 밖으로 나갈 수 있다.
- =length=와 index가 =int=라 대형 collection에서 truncation과 음수 규칙이 섞인다.
- =sublist=는 새 object를 allocation하지만 이름과 =[start,end)= 범위가 불분명하다.

* 업계 collection API와 직접 비교

- [[https://docs.gtk.org/glib/struct.Array.html][GLib =GArray= 공식 API]]는 element storage가 append에 따라 이동할 수 있음을 명시하고
  =len=, append/insert/remove/range, copy, ref/unref와 steal을 서로 다른 operation으로
  노출한다. =coord_list_get/front/back=도 borrowed pointer의 유효 기간을 다음 mutation까지로
  고정하거나, caller-owned =fetch=를 canonical access로 삼아야 한다.
- GLib의 remove와 fast-remove가 순서 보존 여부를 함수 수준에서 구분하는 것처럼
  =remove_value=는 첫 항목/전 항목, stable order와 제거 count를 숨기지 않아야 한다.
- [[https://registry.khronos.org/vulkan/specs/latest/html/vkspec.html][Vulkan C API]]의 count-query/caller-buffer 관행을 따라 bulk export는 필요한 count와
  실제 작성 count, capacity 부족 상태를 제공한다. 새 array를 묵시적으로 할당해 ownership을
  caller에게 전가하지 않는다.

* 목표 계약

** lifetime과 invalidation

- =peek_at/front/back=이 borrowed pointer를 제공한다면 list의 다음 non-const operation,
  destroy 전까지만 유효함을 모든 함수에 명시한다.
- 안전한 기본은 =fetch=로 =coord_t=를 caller storage에 복사하는 것이다. wrapper는
  borrowed pointer 대신 fetch를 사용한다.
- read-only 동시 호출만 허용하고 mutation/destroy와의 동시 실행은 외부 동기화가
  필요함을 metadata에 기록한다.

** index, range와 mutation

- canonical count/index는 =size_t=다. signed Python index 지원은 wrapper에서 normalize한
  뒤 C API에 전달한다.
- =insert(index)==size=는 append로 허용하고 =index>size=는 invalid argument로 고정한다.
- slice는 =[begin,end)=, =begin <= end <= size=, 새 owned list 반환을 이름에 드러낸다.
- =remove_value_ex=는 첫 일치 하나만 제거하고 =out_removed=를 반환하거나,
  =remove_all=과 명시적으로 분리한다.

** allocation과 output guarantee

- 모든 allocation/증가 함수는 exception을 catch하고 status를 반환한다.
- 실패 시 list content와 out parameter를 보존한다.
- bulk 이동에는 =reserve=와 two-call =export=를 제공해 element-by-element wrapper 호출과
  borrowed lifetime 의존을 제거한다.

empty pop은 status 후보를 공통 enum에서 =NOT_FOUND=로 추가할지 =INCOMPLETE=로 사용할지

* wrapper, build와 test 영향

- 현재 Python wrapper의 =get/front/back= child는 parent를 retain하지 않는다. canonical
  wrapper는 value copy를 반환하고 closed-state를 검사한다.
- native =pop=이 value를 반환하므로 Python의 valid =(0,0)=와 empty를 구분할 수 없다.
  새 =try_pop=으로만 Python protocol을 구현한다.
- mutation 중 borrowed child의 UAF/잘못된 값 관찰을 sanitizer test에 포함한다.
- static coord module과 root shared가 동일 =coord_list.cpp=를 포함하고 새 export를
  모두 제공해야 한다.
