* 문서 책임

이 문서는 Navsys 43개 header가 공통으로 사용할 업계 비교 축과 목표 C ABI vocabulary를
정한다. 개별 함수의 최종 변경은 각 header TODO가 현재 구현, consumer와 algorithm
계약을 근거로 결정한다. "업계 평균"을 이름 취향으로 해석하지 않고 다음 관찰 가능한
특성으로 분해한다.

- object layout 은 opaque handle 뒤에 숨기고 lifecycle을 API로 제공한다.
- 정상 결과, 진행 중, 부분 결과, no-path, cancellation, limit와 runtime failure를
  명시적 status로 구분한다.
- callback과 userdata, allocator와 context는 원자적으로 bind하고 retention/thread를
  명시한다.
- caller buffer는 capacity, 실제/필요 count와 incomplete 상태를 함께 제공한다.
- algorithm precondition, optimality/completeness, heuristic, cost 단위와 resource limit을
  선언한다.
- public enum은 숫자와 실제 지원 범위를 고정하고 support query를 제공한다.
- C API의 source/ABI compatibility와 algorithm correctness를 별도 gate로 검증한다.

* 비교 기준

** Production API와 library

- [[https://github.com/recastnavigation/recastnavigation][Recast Navigation 공식 저장소]]:
  runtime query, navigation data, crowd와 debug 책임을 module로 분리한다. Detour query는
  명시적 filter, caller buffer, maximum count와 bitwise =dtStatus=의 success,
  in-progress, partial/buffer-too-small/out-of-nodes 상태를 사용한다.
- [[https://www.boost.org/doc/libs/latest/libs/graph/doc/astar_search.html][Boost.Graph A* 공식 문서]]:
  graph, weight, heuristic, predecessor/distance/color map과 visitor를 분리하고 non-negative
  edge, heuristic 및 complexity precondition을 공개 계약으로 둔다.
- [[https://www.sqlite.org/c3ref/sqlite3.html][SQLite opaque connection]]와
  [[https://www.sqlite.org/c3ref/stmt.html][statement lifecycle]]: 공개 object를 opaque type으로
  만들고 create/bind/step/reset/finalize 상태 전이와 NULL destroy 동작을 문서화한다.
- [[https://docs.libuv.org/en/v1.x/design.html][libuv 공식 design]]: long-lived handle과
  short-lived request를 구분하고 close/callback 이후에만 storage를 회수하는 lifetime과
  cancellation 완료 경로를 명시한다.
- [[https://registry.khronos.org/vulkan/specs/latest/html/vkspec.html][Vulkan 공식 C API spec]]:
  opaque handle, create/destroy, explicit result code, allocator binding, count query 후 caller
  buffer 채우기와 =INCOMPLETE=를 일관된 패턴으로 제공한다.

위 API를 그대로 복제하지 않는다. BYUL naming과 중앙 header 규칙을 유지하면서 외부
consumer가 lifecycle, error와 buffer 계약을 선언만으로 판별할 수 있는 수준을 기준으로
삼는다.

* 현재 Navsys와 기준의 차이

- resource struct layout과 owning pointer가 SDK header에 공개된다.
- 여러 operation이 =route_t*= 또는 =bool=/=void= 하나로 no-path, invalid input, OOM,
  cancellation과 unsupported를 합친다.
- cost/heuristic function과 userdata를 별도 setter로 변경할 수 있어 partial binding이 생긴다.
- route-finder implementation은 저장한 userdata 대신 =nullptr=를 algorithm callback에
  전달한다.
- =coord_hash_values=, string 변환과 collection 반환 allocation에 통일된 BYUL destroy
  경로가 없다.
- enum은 implicit value가 많고 dispatcher가 선언된 algorithm 전부를 지원하지 않는다.
- public macro =MAX_STEP=, =MAX_RADIUS=, =MAX_RETRY=, =DIAGONAL_COST=가 namespace와
  runtime configuration을 대신한다.
- cancellation, resource limit, partial route, callback failure와 output 보존 조건이
  header마다 다르거나 관찰 불가능하다.
- wrapper는 public C header 41개 중 일부만 등록하고 handwritten 선언과 native symbol이
  어긋난다.

* 목표 공통 C ABI Vocabulary

아래 선언은 공통 decision record
[[file:navsys-abi-vocabulary.json][navsys-abi-vocabulary.json]]가 승인한 방향을 사람이
검토할 수 있게 표현한다. 실제 public header와 symbol은 이 문서의 1단계를 완료하기
전에는 추가하지 않는다.

공통 type을 기존 algorithm/component header 중 하나에 임의로 넣지 않는다. 다음 ownership
후보를 비교하고 include DAG가 가장 작은 foundation을 선택한다.

| 후보 | 판정 | 근거 |
|------+------+------|
| 새 =byul/navsys/navsys_status.h= | 승인 | Navsys public-foundation으로 범위를 제한하고 모든 component가 동일 status를 사용한다. |
| 새 =byul/byul_status.h= | 기각 | Navsys 단계가 root 전체 ABI 정책을 선점한다. |
| family별 status enum | 기각 | wrapper mapping과 cross-family operation이 다시 분열된다. |

새 foundation header를 선택하면 최초 43개 inventory의 "누락"으로 취급하지 말고
=generated-by-refactor= row로 manifest에 추가한다. status/options/stats type을
=navsys.h= aggregate에 직접 선언해 module-entry와 foundation 역할을 다시 섞지 않는다.

승인 include DAG는 =navsys.h -> navsys_status.h=,
=public component -> navsys_status.h=이며 =navsys_status.h=는 다른 Navsys component를
include하지 않는다. 이번 단계에서는 owner와 DAG만 승인하고 header는 아직 만들지 않는다.

** C symbol namespace 결정

SQLite의 =sqlite3_=, libuv의 =uv_=와 Vulkan의 =vk=처럼 C ABI는 linker 전역 namespace를
공유한다. BYUL의 기존 =coord_=, =route_=, =maze_= symbol을 즉시 rename하면 source와
binary compatibility가 모두 깨지지만, 새 API에서 =byul_maze_=, =navsys_maze_=와
=maze_=를 파일마다 섞는 것도 허용하지 않는다.

ABI 1.x에서는 기존 대표 type/family prefix를 canonical로 유지하고
=coord_/route_/maze_/route_finder_/dstar_lite_= 계열을 계속 사용한다. 신규 함수에
=navsys_<type>_= 또는 =byul_navsys_<type>_=를 추가하지 않는다. 단,
=navsys_status_t=와 =NAVSYS_STATUS_*=는 function symbol이 아닌 module-foundation
type/enum 이름이므로 예외다.

기존 =move_to/get_changed_coord/default_cost/is_coord_blocked_navgrid= 같은 일반 helper
symbol은 ABI 1.x에서 즉시 삭제하지 않는다. 담당 child가 family-prefixed replacement와
consumer 근거를 제공하면 deprecated forwarding하고, 공개 소비 근거가 없으면
internalize한다. 제거는 ABI-major에서만 허용한다. 개별 TODO의 =byul_*= 또는
=navsys_*= target 초안은 이 승인 정책에 맞게 후속 단계에서 정정한다.

** Status

#+begin_src c
typedef enum e_navsys_status {
    NAVSYS_STATUS_OK = 0,
    NAVSYS_STATUS_INVALID_ARGUMENT = -1,
    NAVSYS_STATUS_OUT_OF_MEMORY = -2,
    NAVSYS_STATUS_UNSUPPORTED = -3,
    NAVSYS_STATUS_CALLBACK_FAILED = -4,
    NAVSYS_STATUS_CORRUPT_STATE = -5,
    NAVSYS_STATUS_NOT_FOUND = -6,
    NAVSYS_STATUS_INVALIDATED = -7,
    NAVSYS_STATUS_NO_PATH = -8,
    NAVSYS_STATUS_CANCELLED = -9,
    NAVSYS_STATUS_LIMIT_REACHED = -10,
    NAVSYS_STATUS_INCOMPLETE = -11,
    NAVSYS_STATUS_IN_PROGRESS = -12
} navsys_status_t;
#+end_src

- 양수는 count/result 전용으로 남기고 =OK=가 아닌 모든 status를 음수로 고정한다.
- =NOT_FOUND=는 정상적인 lookup miss, =INVALIDATED=는 iterator/view 세대 불일치로
  분리한다.
- 비성공 status의 usable/partial output 여부를 각 함수가 명시한다.
- negative error에서 기존 object와 out parameter가 보존되는지 명시한다.
- thread-local/global last-error를 유일한 실패 경로로 사용하지 않는다.
- 세부 진단 문자열은 caller-buffer query 또는 immutable static string으로 제공한다.

** Opaque resource와 lifecycle

#+begin_src c
typedef struct s_route_finder route_finder_t;

BYUL_API route_finder_t* route_finder_create(
    const route_finder_create_info_t* create_info);

BYUL_API void route_finder_destroy(route_finder_t* finder);

BYUL_API navsys_status_t route_finder_reset(route_finder_t* finder);
#+end_src

- BYUL 규칙에 따라 allocation을 반환하는 =create/copy=와 =destroy=를 짝짓는다.
- =destroy(NULL)= 허용 여부를 모든 resource family에서 통일한다.
- =reset=은 configuration/callback/binding 중 무엇을 보존하는지 명시한다.
- caller storage =init/free= family가 필요한 type은 opaque heap family와 섞지 않는다.
- wrapper/SDK ABI 확인용 size/align query는 storage allocation API로 오용하지 않는다.

** Callback binding

#+begin_src c

- 비용 callback은 음수, NaN, Inf와 failure를 정상 cost와 구분한다.
- callback 실행 중 unbind/destroy 가능 여부를 명시하고 test한다.

** Caller buffer와 collection export

#+begin_src c
BYUL_API navsys_status_t route_export_coords(
    const route_t* route,
    coord_t* out_coords,
    size_t capacity,
    size_t* out_count);
#+end_src

- =out_coords == NULL=과 =capacity == 0=으로 필요한 count를 조회하는지 명시한다.
- 작은 buffer는 쓰인 count/필요 count 중 하나의 일관된 의미와 =INCOMPLETE=를 반환한다.
- element 단위와 byte 단위를 parameter 이름으로 구분한다.
- internal container pointer를 반환하지 않고 caller buffer, immutable view 또는 snapshot을
  선택한다.

** Allocator와 workspace

- library heap allocation은 항상 대응 BYUL destroy로 회수한다.
- 고빈도 search는 caller-provided workspace 또는 create-time allocator binding을 검토한다.
- allocator function과 userdata를 원자적인 immutable binding으로 다룬다.
- allocation alignment, zero-size, realloc semantics와 callback thread를 정의한다.
- peak/open-set/closed-set memory와 caller workspace 사용량을 stats 또는 memory query로
  관찰 가능하게 한다.

** Enum과 capability

#+begin_src c

* 호환성 단계

| 단계 | 허용 변경 | 금지 변경 |
|------+-----------+-----------|
| ABI 1.x additive | 새 status/options/bind/export/query symbol, metadata, test | 기존 symbol 삭제, enum renumber, struct layout 변경 |
