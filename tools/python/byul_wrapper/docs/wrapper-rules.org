#+TITLE: BYUL Python Wrapper 자동 생성 규칙
#+LANGUAGE: ko
#+OPTIONS: toc:3 num:t

* 문서 목적

이 문서는 BYUL SDK의 public C header로부터 Python CFFI wrapper를 자동 생성하기
위한 단일 규칙 문서다.

자동 생성 범위는 C ABI type에 직접 대응하는 저수준 =c_*= class까지다. 고수준 값
객체, facade와 application domain object는 별도 package에서 작성한다.

Public header 자체의 기준은 다음 두 문서로 분리한다.

- [[file:../../../../docs/ko/byul-sdk/header-declaration-rules.org][BYUL Public C Header 선언 규칙]]
- [[file:../../../../docs/ko/byul-sdk/header-comment-rules.org][BYUL Public C Header 코멘트 규칙]]

- MUST/MUST NOT 위반은 생성을 실패시킨다.
- SHOULD는 예외 근거가 없으면 적용한다.
- MAY는 선택 사항이다.

* 핵심 원칙

** SDK가 유일한 입력이다

Wrapper generator는 source tree를 읽지 않고, 압축을 푼 SDK root 또는
=byul-sdk-*.zip=만 입력으로 받는다.

SDK 기본 구조는 다음과 같다.

#+begin_example
byul/
  bin/byul.dll
  include/byul/*.h
  lib/byul/byul.lib
#+end_example

다른 platform의 shared library 위치는 해당 SDK layout을 따른다.

** Header 선언이 기계 판독 계약이다

Generator는 일반 설명 comment에서 소유권이나 Python 이름을 추론하지 않는다.

판단에는 파일명, type과 struct 선언, 함수명, 반환형, parameter, =const= 및
구조화된 =@byul.*= metadata만 사용한다. 일반 설명문에서 의미를 추론하지 않는다.

** C 함수와 Python API의 대응은 결정적이어야 한다

같은 C 선언은 언제나 같은 Python 코드를 생성해야 한다. Generator가 문맥이나
추측에 따라 다른 결과를 만들면 안 된다.

각 Python property와 method에는 원본 C symbol 추적 정보를 남긴다.

* Header와 type 명명 규칙

** 대표 type

Header file stem과 대표 C type 이름은 일치해야 한다.

| Header | 대표 type | Python class |
|--------+-----------+--------------|
| coord.h | coord_t | c_coord |
| coord_list.h | coord_list_t | c_coord_list |
| navgrid.h | navgrid_t | c_navgrid |
| route.h | route_t | c_route |
| route_finder.h | route_finder_t | c_route_finder |

규칙은 다음과 같다.

#+begin_example
<name>.h -> <name>_t -> c_<name> -> <name>.py
#+end_example

Header에 대표 type이 없고 free function만 있다면 module function 생성 대상으로
분류한다.

Type의 storage, ABI layout, allocation 검증은
[[file:../../../../docs/ko/byul-sdk/header-declaration-rules.org][Header 선언 규칙]]의 =Type 저장 방식 분류=와
=ABI 크기와 layout 정보=를 우선 적용한다.

Generator는 SDK의 =sizeof/alignof/offsetof= 정보와 runtime DLL의 ABI fingerprint를
검증한 뒤에만 wrapper를 import 가능 상태로 만든다.

** Opaque type과 public value struct

Opaque type:

#+begin_src c
typedef struct s_coord_list coord_list_t;
#+end_src

Public value struct:

#+begin_src c
typedef struct s_coord {
    int x;
    int y;
} coord_t;
#+end_src

두 형식 모두 =c_*= class를 만들 수 있다. 다만 public field에 직접 접근하는
Python property는 자동 생성하지 않는다. Public getter/setter 함수가 있을 때만
property를 생성한다. DLL ABI를 우회하여 CFFI field에 직접 접근하지 않는 것이
기본 원칙이다.

** 함수 prefix

대표 type의 함수는 반드시 다음 prefix를 사용한다.

#+begin_example
<type-name>_<operation>
#+end_example

예:

#+begin_example
coord_create
coord_get_x
coord_set_x
coord_distance
coord_list_push_back
route_finder_run
#+end_example

Prefix가 대표 type과 일치하지 않으면 자동 instance method 생성 대상이 아니다.

* Python 이름 변환 규칙

** 일반 instance method

함수의 첫 번째 parameter가 대표 type pointer이고 함수명이 대표 type prefix로
시작하면 instance method로 변환한다.

#+begin_src c
float coord_distance(const coord_t* a, const coord_t* b);
#+end_src

변환:

#+begin_src python
class c_coord:
    def distance(self, other):
        return C.coord_distance(self._c, other._c)
#+end_src

Python method 이름은 C 함수에서 정확히 한 번 대표 type prefix를 제거한 이름이다.

#+begin_example
coord_distance           -> distance
coord_degree             -> degree
coord_init_full          -> init_full
coord_list_push_back     -> push_back
navgrid_block_coord      -> block_coord
dstar_lite_update_vertex -> update_vertex
#+end_example

함수 중간이나 suffix를 임의로 축약하지 않는다.

** Property getter

다음 조건을 모두 만족하면 property getter로 변환한다.

1. 함수명이 =<type>_get_<property>= 형식이다.
2. 첫 parameter가 =const <type>_t*= 또는 =<type>_t*= 다.
3. 첫 parameter 외에 입력 parameter가 없다.
4. 반환형이 =void= 가 아니다.

#+begin_src c
int coord_get_x(const coord_t* coord);
#+end_src

#+begin_src python
@property
def x(self):
    return C.coord_get_x(self._c)
#+end_src

** Property setter

다음 조건을 모두 만족하면 property setter로 변환한다.

1. 함수명이 =<type>_set_<property>= 형식이다.
2. 첫 parameter가 non-const =<type>_t*= 다.
3. 첫 parameter 외에 값 parameter가 정확히 하나다.
4. 반환형이 =void= 이거나 프로젝트의 통일된 status 반환형이다.

#+begin_src c
void coord_set_x(coord_t* coord, int value);
#+end_src

#+begin_src python
@x.setter
def x(self, value):
    C.coord_set_x(self._c, value)
#+end_src

Getter만 있으면 read-only property를 생성한다. Setter만 있는 API는 property로
만들지 않고 =set_<property>()= method로 남긴다.

** Parameter가 있는 get 함수

=get= 함수라도 첫 객체 parameter 외에 조회 key/index parameter가 있으면
property가 아니다.

#+begin_src c
const coord_t* coord_list_get(
    const coord_list_t* list,
    int index);
#+end_src

#+begin_src python
def get(self, index):
    ...
#+end_src

=__getitem__= 같은 Python protocol은 별도의 고정 mapping 규칙이 있을 때만
추가한다. 기본 method =get()=을 제거하면 안 된다.

** Boolean query

다음 prefix는 =bool= 반환 method로 그대로 변환한다.

#+begin_example
is_<state>
has_<item>
contains
empty
equal
#+end_example

반환값은 =bool(...)= 로 정규화한다.

** Python special method

다음 mapping만 허용한다.

| C suffix | Python special method |
|----------+-----------------------|
| equal | __eq__ |
| hash | __hash__ |
| add | __add__ |
| sub | __sub__ |
| mul | __mul__ |
| div | __floordiv__ 또는 명시된 division 규칙 |

Special method를 생성하더라도 원래 C 함수와 대응 관계를 docstring 또는 generated
metadata에 남긴다. 의미가 Python protocol과 정확히 일치하지 않으면 special
method를 생성하지 않고 일반 method로 둔다.

* 생성과 해제 규칙

** create

#+begin_example
<type>_create
<type>_create_<variant>
#+end_example

=create= 계열 pointer 반환은 새 native object를 할당한다.

- 반환 wrapper는 =own=True= 다.
- NULL이면 allocation 실패로 처리한다.
- 대응하는 =<type>_destroy=가 반드시 존재해야 한다.
- finalizer는 정확히 해당 destroy symbol을 사용한다.

기본 create와 full create는 Python =__init__=으로 합칠 수 있다.

#+begin_src c
coord_t* coord_create(void);
coord_t* coord_create_full(int x, int y);
#+end_src

#+begin_src python
class c_coord:
    def __init__(self, x=0, y=0):
        self._c = C.coord_create_full(x, y)
        self._own = True
#+end_src

생성 variant들의 parameter 구조가 하나의 =__init__=으로 명확히 합쳐지지 않으면
classmethod를 생성한다.

** copy와 clone

#+begin_example
<type>_copy
<type>_copy_<item>
<type>_clone_<item>
#+end_example

=copy=와 =clone= 계열 pointer 반환은 새 allocation이다.

- 반환 wrapper는 =own=True= 다.
- 결과가 NULL일 수 있으면 =None= 또는 통일된 예외 규칙을 적용한다.
- caller가 대응 destroy를 호출한다.

** destroy

#+begin_src c
void coord_destroy(coord_t* coord);
#+end_src

Destroy 함수는 Python에서 다음 수명 관리에 연결한다.

- =weakref.finalize=
- =close()=
- =__del__()=
- =__enter__()/__exit__()=

Public Python method 이름을 =coord_destroy=로 유지할 필요는 없지만 generated
metadata에는 destroy symbol을 기록한다.

=close()=는 여러 번 호출해도 native destroy를 한 번만 호출해야 한다.

** free

=free=는 object memory 자체가 아니라 내부 resource만 정리하는 API에 사용한다.

#+begin_example
<type>_free   -> 내부 resource 해제, struct memory 유지
<type>_destroy -> 내부 resource와 object memory 해제
#+end_example

두 함수의 의미를 섞어 쓰면 안 된다.

** clear와 reset

- =clear=: collection 또는 contents를 비우고 object identity와 configuration을 유지
- =reset=: object를 초기 상태로 되돌림

두 함수는 native object를 파괴하지 않으므로 =_own= 상태를 변경하지 않는다.

* Pointer 반환과 소유권 규칙

** get과 peek

#+begin_example
<type>_get_<item>
<type>_peek_<item>
#+end_example

=get=과 =peek=의 pointer 반환은 새 allocation이 아니다.

- 반환 wrapper는 =own=False= 다.
- 반환 pointer가 속한 parent wrapper를 =_owner=로 보관한다.
- 반환 wrapper close는 native destroy를 호출하지 않는다.

#+begin_src python
result = c_coord(raw_ptr=ptr, own=False)
result._owner = self
#+end_src

=borrowed=는 Python API 이름이 아니다. Wrapper 내부 메모리 상태를 설명하는
용어일 뿐이다. Public method 이름은 prefix 제거 규칙에 따라 =get()= 또는
=peek_<item>()=이다.

** fetch

=fetch=는 caller가 제공한 out parameter에 값을 복사한다.

#+begin_src c
bool route_finder_fetch_start(
    const route_finder_t* finder,
    coord_t* out);
#+end_src

Generator는 out value를 준비하고 native 함수 실행 후 Python wrapper 또는 기본형으로
반환할 수 있다. =fetch=는 내부 pointer를 노출하지 않는다.

** take

=take=는 caller가 가진 object의 소유권을 callee에게 넘긴다.

#+begin_example
<type>_take_<item>
#+end_example

호출 성공 후 전달한 Python wrapper는 더 이상 destroy하지 않아야 한다. Generator는
해당 wrapper의 finalizer를 detach하고 closed/released 상태로 전환한다.

** release

=release=는 object가 소유하던 native pointer를 caller에게 넘긴다.

#+begin_example
<type>_release_<item>
#+end_example

반환 pointer wrapper는 =own=True= 다. 원래 parent는 해당 pointer를 더 이상
파괴하지 않는다.

** bind와 unbind

=bind=는 pointer 또는 callback을 나중에 사용하기 위해 non-owning reference로
보관한다.

#+begin_example
<type>_bind_<item>
<type>_unbind_<item>
#+end_example

Python wrapper는 bind된 Python object 또는 callback을 native object 수명 동안
강한 참조로 보관한다. =unbind= 또는 destroy 시 참조를 제거한다.

** set

=set=은 전달받은 값을 즉시 복사한다. Pointer를 장기간 보관하거나 소유권을
가져가는 함수에 =set= 이름을 사용하면 안 된다.

* 반환형과 NULL 규칙

** Pointer

Pointer 반환에서 NULL 처리는 함수 계열에 따라 결정한다.

- =create/copy/clone=: allocation 또는 operation 실패
- =get/peek/find=: 값 없음
- =release=: 소유한 값 없음 또는 실패

Wrapper 전체에서 =None=과 exception 사용 기준을 통일해야 한다.

** bool

C =bool=은 Python =bool=로 변환한다.

#+begin_src python
return bool(C.navgrid_is_inside(self._c, x, y))
#+end_src

** Status int와 count int

두 종류를 함수 이름 또는 프로젝트 전역 규칙으로 구분해야 한다.

- status: 0 성공, 음수 또는 nonzero 실패
- count/index: 0 이상 정상, 음수 오류 또는 not-found

동일한 naming family에서 규칙이 달라지면 자동 생성을 거부한다.

** Enum

C enum 반환은 대응 Python =IntEnum=으로 변환한다. 알 수 없는 정수 값의 처리
방식을 wrapper 전체에서 통일한다.

** Struct value

Pointer가 아닌 struct value 반환은 allocation으로 간주하지 않는다. 필요한 경우
값을 새 caller-owned wrapper storage로 복사하되 원래 반환값에 destroy를 적용하지
않는다.

* Callback 규칙

** callback type 명명

Callback typedef는 =<purpose>_func= 또는 프로젝트가 정한 단일 suffix를 사용한다.

Callback 반환 pointer의 메모리 의미도 이름에 들어가야 한다.

#+begin_example
changed_coords_copy_func -> callback이 새 object 반환, C가 destroy
changed_coords_get_func  -> callback이 기존 object 반환, C가 destroy하지 않음
#+end_example

의미를 판단할 수 없는 =changed_coords_func= 같은 이름은 자동 생성 lint에서
거부한다.

** callback 등록

#+begin_example
<type>_bind_<purpose>_callback
<type>_unbind_<purpose>_callback
#+end_example

=bind=된 callback은 native object가 unbind 또는 destroy될 때까지 사용될 수 있다.
Python wrapper는 다음을 멤버로 보관해야 한다.

- 원본 Python callable
- =ffi.callback= object
- userdata object
- =ffi.new_handle= 반환 object

** callback 호출 thread

BYUL ABI는 callback thread 정책을 전역으로 통일해야 한다. 권장 규칙은
"native 함수를 실행한 thread에서 동기 호출"이다.

별도 worker thread callback이 필요한 경우 API family 이름 또는 별도 구조적
선언으로 구분해야 하며 Python wrapper는 GIL, exception, 종료 경쟁을 테스트해야
한다.

** callback exception

Python exception이 C 경계를 넘어가면 안 된다. Generated callback은 exception을
잡고 함수 반환형에 맞는 안전한 실패값을 반환한다. 동시에 traceback을 logging
또는 저장된 error channel로 전달한다.

* 문자열 규칙

문자열 함수는 allocation을 이름으로 구분한다.

#+begin_example
<type>_get_<name>_string   -> static 또는 내부 문자열, 해제 금지
<type>_copy_<name>_string  -> 새 문자열, 전용 destroy 필요
<type>_fetch_<name>_string -> caller buffer에 복사
#+end_example

모든 public 문자열 encoding은 UTF-8로 통일하는 것을 원칙으로 한다.

Owned =char*= 반환에는 반드시 하나의 통일된 문자열 destroy 함수가 있어야 한다.

* Array와 buffer 규칙

Caller buffer 출력 함수는 다음 parameter 순서를 사용한다.

#+begin_src c
int type_fetch_items(
    const type_t* object,
    item_t* out_items,
    int capacity);
#+end_src

반환값은 실제 작성 개수이며 음수는 오류다. 필요한 크기 조회가 필요하면
=type_get_item_count()=를 별도로 제공한다.

Pointer와 길이를 반환하는 임의의 이중 pointer API는 자동 생성 난도가 높으므로
가능하면 fetch buffer 형태로 통일한다.

* Container 규칙

** length와 empty

#+begin_example
<type>_length -> __len__ 추가 가능, length()도 유지 가능
<type>_empty  -> bool method
#+end_example

** get과 Python indexing

=get(index)=는 기본 생성 method다. =__getitem__=은 index 범위와 error 규칙이
명확할 때 보조 protocol로 생성할 수 있다.

** push/pop

- =push_*=는 값을 container에 복사한다.
- Pointer 소유권을 가져가면 =take_*=를 사용한다.
- =pop= 반환이 value인지 owned pointer인지 반환형과 이름으로 명확해야 한다.

* CFFI 선언 생성 규칙

** Preprocessor 처리

Generator는 다음을 제거하거나 정규화한다.

- include guard
- =#include=
- platform export macro =BYUL_API=
- =extern "C"= block
- CFFI가 해석하지 못하는 compiler attribute

SDK header dependency는 include graph 또는 flat SDK include directory를 기준으로
해결한다.

** Generated 영역

Python module의 generated 영역은 사람이 수정하지 않는다.

#+begin_src python
ffi.cdef("""
/* generated: do not edit */
...
""")
#+end_src

Generator를 다시 실행했을 때 byte-for-byte 동일한 결과가 나와야 한다.

** 원본 추적

Generated block에는 다음 정보를 기록해야 한다.

- SDK version
- SDK platform/toolchain/configuration
- header 상대 경로
- header content hash
- generator version

* SDK 검증 규칙

Wrapper 생성 전에 다음을 검사한다.

1. SDK root 구조가 올바른가.
2. shared library가 존재하는가.
3. =include/byul=이 존재하는가.
4. SDK version이 지원 범위인가.
5. header hash와 생성 기록이 일치하는가.
6. DLL export symbol과 public 함수 선언이 일치하는가.
7. Debug/Release 및 architecture가 현재 Python 환경과 맞는가.

오래된 SDK에서 wrapper를 생성하면 안 된다. SDK의 header와 DLL이 서로 다른
build에서 나온 경우 즉시 실패해야 한다.

* ABI linter 규칙

Generator 실행 전 linter가 public header 전체를 검사한다.

** Type 검사

- =foo.h=의 대표 type이 =foo_t=인지 확인한다.
- opaque/public struct 선언이 중복되지 않는지 확인한다.
- 대표 type에 create가 있으면 destroy가 있는지 확인한다.

** 함수 이름 검사

Pointer 반환 함수는 다음 allocation family 중 하나로 분류 가능해야 한다.

#+begin_example
create copy clone get peek fetch take release bind
#+end_example

분류할 수 없는 pointer 반환 함수는 생성 실패다.

** get/set property 검사

- getter와 setter property 이름이 같은가.
- getter 반환형과 setter 값 타입이 같은가.
- getter가 추가 입력 parameter를 갖지 않는가.
- setter가 값 parameter를 하나만 갖는가.

** Callback 검사

- callback 반환 pointer의 copy/get 의미가 이름에 있는가.
- bind callback에 대응 unbind 또는 destroy 수명 끝점이 있는가.
- userdata parameter naming이 통일됐는가.

** 문자열과 buffer 검사

- =char*= 반환이 get/copy/fetch family로 분류되는가.
- owned 문자열 destroy 함수가 있는가.
- buffer 출력 함수에 capacity가 있는가.

* Wrapper code 생성 절차

Generator는 다음 순서를 따른다.

1. SDK를 열고 metadata, header와 library를 검증한다.
2. C 선언 AST를 만들고 ABI linter를 실행한다.
3. 대표 type과 function family를 분류한다.
4. =ffi.cdef()=와 =c_*= class를 생성한다.
5. Formatting, DLL symbol과 generated test를 검증한다.
6. =--check=에서는 파일을 수정하지 않고 차이만 보고한다.

* Test 생성 규칙

** 모든 object type

- create가 NULL이 아닌지 검사
- destroy가 한 번만 호출되는지 검사
- =close()= 두 번 호출이 안전한지 검사
- context manager 종료가 destroy하는지 검사
- close 후 method 접근을 차단하는지 검사

** copy type

- copy가 다른 pointer인지 검사
- 값이 같은지 검사
- 한쪽 수정이 다른 쪽에 영향을 주지 않는지 검사
- 각각 독립적으로 destroy 가능한지 검사

** get/peek pointer

- 반환 wrapper가 =own=False=인지 검사
- child close가 native pointer를 destroy하지 않는지 검사
- parent reference가 유지되는지 검사

** property

- getter 결과가 C 함수 결과와 같은지 검사
- setter 후 getter가 변경값을 반환하는지 검사
- read-only property에 setter가 생성되지 않는지 검사

** callback

- Python callback이 호출되는지 검사
- userdata identity가 유지되는지 검사
- 강제 GC 후에도 callback이 유효한지 검사
- unbind 후 호출되지 않는지 검사
- callback exception이 C 경계를 넘지 않는지 검사

** 반복 수명 검사

생성/복사/해제를 반복하여 memory 증가와 double-free를 검사한다. Native Debug
build에서는 AddressSanitizer 또는 platform memory diagnostic을 함께 사용한다.

* 수동 코드와 예외 처리

규칙으로 생성할 수 없는 API를 generator가 추측해서는 안 된다.

예외 API는 다음 중 하나를 선택해야 한다.

1. C ABI 이름을 규칙에 맞게 변경한다.
2. 자동 생성 대상에서 제외하고 명시적인 manual wrapper registry에 등록한다.

Manual registry에는 반드시 이유와 제거 계획을 기록한다. Comment 한 줄로 생성
규칙을 임시 우회하는 방식은 사용하지 않는다.

* 호환성 정책

** C ABI 변경

Public 함수명, parameter, 반환형, struct layout 변경은 ABI 변경이다. SDK version과
wrapper generated hash를 함께 갱신해야 한다.

** Python wrapper 변경

C 함수 이름 규칙에서 결정되는 Python property/method 이름도 public API다. Naming
rule 변경 시 전체 generated API가 바뀔 수 있으므로 generator version을 올리고
migration을 기록한다.

** 기존 수동 wrapper 전환

Generated wrapper가 기존 test와 memory diagnostic을 통과한 뒤 consumer를 전환하고
handwritten 영역을 제거한다.
